# 追更详情页分集虚拟列表设计

## 概要

将“我的追更”详情页中的分集区域从当前 `QScrollArea + QHBoxLayout + QWidget` 卡片列表，重构为基于 Qt model/view 的虚拟列表结构。

目标：

- 支持多季内容，例如 23 季剧集。
- 支持单季超长内容，例如 1 季 1200 集动画。
- 保持以“浏览分集标题和更新时间后再选”为主的使用方式。
- 保留分集封面能力，但不回退到大卡片全量渲染。
- 提供三种可切换显示模式：`简洁`、`封面`、`完整`。
- 切换季、切换模式、图片异步加载时都不重建整页，不引入 1200 个独立分集 widget。

## 变更范围

主要涉及：

- `src/atv_player/ui/following_detail_page.py`
- 新增或内嵌的分集 view/model/delegate 实现
- `tests/test_following_detail_page_ui.py`

不涉及：

- `FollowingController` 的详情抓取逻辑
- `FollowingEpisode` / `FollowingDetailSnapshot` 的基础字段结构
- 分集详情弹窗的数据来源
- 页面顶部元数据区、海报轮播区、演职员区的整体布局重做

## 现状问题

当前分集区实现方式：

- 用 `QScrollArea` 承载 `_episodes_container`
- 用 `QHBoxLayout` 把每个 `FollowingEpisodeCard` 依次加进去
- 即使做了分批渲染，最终仍会创建并保留所有分集 `QWidget`

这在以下场景下会持续恶化：

- 单季 1200 集时，widget 数量、布局计算、图片加载线程数都会膨胀
- 多季场景只能展示一个扁平分集集合，没有清晰季导航
- 不同信息密度需求只能靠重新设计卡片，难以扩展成轻量模式

根因不是“首批渲染速度”，而是“最终仍然全量持有所有分集 widget”。

## 设计目标

### 功能目标

- 用户可以在左侧快速切换季
- 用户可以在右侧连续浏览当前季所有分集
- 用户可以根据偏好在三种显示模式之间切换
- 用户点击任一分集后，继续打开现有分集详情弹窗

### 性能目标

- 当前季即使有 1200 集，也只渲染可视区域附近的少量列表项
- 模式切换只触发列表重绘，不销毁和重建整页
- 图片异步返回时只刷新受影响列表项，不触发布局级抖动

### 交互目标

- 多季剧集结构清晰
- 单季长番滚动连续，不做区间分页或区间切换
- 默认体验兼顾封面和浏览效率

## 总体方案

分集区改成两栏结构：

- 左侧：季导航列表
- 右侧：当前季分集虚拟列表

实现路径：

- 左侧使用 `QListView` 展示季列表
- 右侧使用 `QListView`
- 数据由 `QAbstractListModel` 提供
- 视觉由 `QStyledItemDelegate` 绘制

整个分集区不再为每一集创建独立 `QPushButton` 或 `QFrame`。

## 布局设计

### 分集区整体布局

保留当前“分集详情”区域作为页面下半部分的一块 panel，但内部拆成：

- 标题栏
- 主体区

标题栏包含：

- 左侧标题：`分集详情`
- 右侧模式切换：`简洁` / `封面` / `完整`

主体区包含：

- 左栏：季导航
- 右栏：当前季分集列表

### 左侧季导航

左侧季导航负责展示：

- 季名称，例如 `第 1 季`
- 集数摘要，例如 `24 集`、`1200 集`
- 当前选中态

行为：

- 点击切换当前季
- 切换季时仅替换右侧当前季数据
- 不影响页面顶部元数据区和演职员区

特殊情况：

- 如果只有一季，左侧仍保留一个季导航项，保持统一结构
- 如果分集数据里没有显式季号，则回退为单季 `第 1 季`

### 右侧分集列表

右侧分集列表是连续纵向虚拟滚动列表。

它必须支持：

- 当前季只有几十集
- 当前季有 1200 集
- 用户上下滚动浏览标题和日期
- 键盘上下移动选中项

不提供：

- 区间分页
- 按 50 集切段切换
- 横向大卡片滚动

## 三种显示模式

三种模式共用同一个 `QListView + model + delegate`，只切换绘制策略，不切换基础控件结构。

### 简洁模式

显示内容：

- `集号 + 标题`
- `更新时间`

不显示：

- 分集封面
- 分集简介

用途：

- 信息密度最高
- 最适合超长分集快速扫读

### 封面模式

显示内容：

- 左侧固定 16:9 小横图
- 右侧第一行：`集号 + 标题`
- 右侧第二行：`更新时间`

不显示：

- 分集简介正文

用途：

- 作为默认推荐模式
- 平衡封面识别和浏览效率

### 完整模式

显示内容：

- 左侧固定 16:9 小横图
- 右侧第一行：`集号 + 标题`
- 右侧第二行：`更新时间`
- 右侧第三块：`简介摘要`

约束：

- 简介最多显示 2 到 3 行
- 必须省略超长文本，不能无限展开
- 行高可以高于前两种模式，但仍要保持固定或离散可控，避免滚动抖动

用途：

- 给需要更多剧情提示的用户
- 不替代点击后的分集详情弹窗

## 分集行设计

### 文本内容

标题主行：

- 优先显示 `集号 + 标题`
- 如果标题为空，则显示 `第 N 集`

副行：

- 显示 `更新时间`
- 如果日期为空，则不显示副行文本，但保持当前模式的既定 item 高度

补充状态：

- 对 `episode_number <= current_episode` 的分集显示 `已看`
- 对 `is_special=True` 的分集显示 `特别篇`

这些状态标记必须是轻量视觉提示，不能挤占标题主体空间。

### 封面缩略图

封面只在 `封面` 与 `完整` 模式出现。

规则：

- 固定 16:9 比例
- 使用统一小尺寸缩略图
- 缺图时显示占位块
- 不因为图片缺失改变行高

### 简介摘要

简介只在 `完整` 模式出现。

规则：

- 基于 `FollowingEpisode.overview`
- 自动截断并省略号结尾
- 保持固定最大行数

完整长简介仍由点击后弹窗承载。

## 点击与键盘交互

单击行为：

- 选中分集项

激活行为：

- 双击
- 回车

激活后：

- 打开现有 `FollowingEpisodePreviewDialog`

键盘交互：

- 上下键移动选中项
- `Enter` / `Return` 打开详情弹窗

## 初始定位与季内状态恢复

页面首次打开时：

- 若存在 `current_episode`，右侧列表优先定位到该集附近
- 不强制把该项顶到视口最顶部，避免失去上下文

季切换时，每季单独保存：

- `scroll_position`
- `current_index`

用户切回某季时：

- 恢复该季上次滚动位置
- 恢复该季上次选中项

这对 23 季长剧尤其重要。

## 数据模型设计

拆成两层模型。

### SeasonListModel

负责左侧季导航。

每项至少包含：

- `season_number`
- `display_title`
- `episode_count`
- `is_current`

职责：

- 提供季列表显示文本
- 提供当前选中季索引
- 不负责分集图片或分集详情数据

### EpisodeListModel

负责右侧当前季分集。

每项至少包含：

- `episode_number`
- `season_number`
- `title`
- `air_date`
- `overview`
- `still`
- `is_special`
- `is_watched`
- `has_update`

说明：

- `is_watched`、`has_update` 可以先由页面层根据 `FollowingRecord.current_episode` 和当前集号推导，不要求后端额外存储
- `EpisodeListModel` 只承载当前季分集，而不是所有季一次性混在一个超长 model 里

### 页面层状态

页面自身需要维护：

- 当前季索引
- 当前显示模式
- 每季滚动位置
- 每季选中项
- 当前季缩略图加载状态缓存

## 分组与季数据整理

`FollowingDetailSnapshot.episodes` 已经包含 `season_number` 字段。

页面加载详情后应先做一次分组整理：

1. 按 `season_number` 分组
2. 如果没有有效季号，则全部归为第 1 季
3. 组内按 `episode_number` 升序排列

这样可以同时覆盖：

- 明确的多季剧集
- 单季长番
- 只有一组 episode 且无季号的旧数据

## 图片加载与缓存

图片加载不能沿用“每行创建一个线程”的策略。

- 把分集缩略图加载抽成共享图片加载器
- 对同一 `still` URL 做缓存
- 对列表只请求缩略图尺寸，不请求大图
- 图片返回后只刷新对应 model index

行为约束：

- 列表初始渲染先显示占位图和文字
- 缩略图异步替换占位图
- 不因图片到达而改变 item 尺寸

非目标：

- 不在列表里加载高清原图
- 不在列表里做复杂动效

## 与现有分集弹窗的关系

现有 `FollowingEpisodePreviewDialog` 保留。

列表职责：

- 高密度浏览
- 快速定位
- 触发选择

弹窗职责：

- 展示大图
- 展示完整日期
- 展示完整剧情简介

这可以避免把“浏览层”和“详情层”混在一处，导致列表模式过重。

## 与现有详情页的迁移边界

本次重构只替换分集区实现，不重写整个详情页。

保留：

- 顶部元数据面板
- 海报轮播面板
- 演职员区域
- 手动检查 / 更新元数据 / 设置进度 / 取消追更等动作

替换：

- `_episodes_container`
- `_episodes_layout`
- `FollowingEpisodeCard`
- `_render_next_batch()` 中分集 widget 的创建逻辑

新增：

- 季导航 view/model
- 分集列表 view/model
- 分集 delegate
- 图片缩略图缓存/加载桥接
- 显示模式切换控件

## 状态持久化

要求：

- 在当前页面生命周期内保留所选显示模式
- 将显示模式提升为应用级 UI 偏好
- 用户下次进入任意追更详情页时沿用上次模式

不要求持久化：

- 每个追更条目的季滚动位置跨应用重启保存

页面内记忆即可。

## 性能约束

验收标准应体现为：

- 打开含 1200 集的详情页时，页面可快速出现，列表先显示文字和占位图
- 滚动过程中不会因为大量 widget 构建导致明显卡顿
- 切换模式不会触发整页闪烁
- 切换季不会导致页面顶部区域重新加载

## 测试策略

先补测试，再实现。

重点覆盖：

- 多季分组后左侧季导航显示正确
- 单季 1200 集时右侧只绑定当前季 model，不创建 `FollowingEpisodeCard`
- 三种模式切换只改变显示模式状态和 delegate 渲染参数
- 模式切换控件默认值正确
- 页面首次打开时能定位到当前观看集附近
- 切换季后恢复该季滚动位置和选中项
- 缺失季号时回退到第 1 季
- 缺失标题、日期、封面、简介时渲染退化正确
- 点击或激活列表项仍能打开分集详情弹窗

测试至少验证：

- 分集区不再实例化旧 `FollowingEpisodeCard`
- 页面通过 model/view 路径渲染分集
- 模式切换和季切换走状态更新而不是整页重建

## 实施建议

建议分阶段落地：

1. 先引入季分组和 `EpisodeListModel`
2. 用 `QListView + delegate` 跑通简洁模式
3. 加入封面模式与缩略图缓存
4. 加入完整模式与简介摘要
5. 接入季状态恢复和显示模式持久化
6. 删除旧分集卡片渲染路径

这样可以控制改动面，并在每一步保持页面可运行。

## 非目标

本次不做：

- 分集区再次回到横向大卡片设计
- 单季 1200 集按区间分页
- 分集简介在列表中无限展开
- 分集封面自动轮播
- 多季与单季混合瀑布流布局
- 重做分集详情弹窗样式
