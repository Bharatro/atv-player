# Player Log Max Height Design

## Summary

播放器右侧详情栏中的 `播放日志` 区域目前没有高度上限，日志变多时会挤压 `影片详情` 区域。这个设计为日志区增加动态最大高度约束：日志整个区域最多占右侧详情栏高度的 `1/4`，超出的内容继续通过现有滚动能力查看。

本次改动只限制高度，不改变日志内容、显示开关、滚动行为或侧栏整体结构。

## Goals

- 让 `播放日志` 区域不会在右侧详情栏中无限扩张。
- 保证日志整个区域最高只占详情栏高度的 `1/4`。
- 保留现有日志滚动查看能力。
- 保持现有 `详情` / `播放日志` 显示与隐藏逻辑不变。

## Non-Goals

- 把详情栏改成新的分栏或嵌套 `QSplitter` 结构。
- 修改日志文案、写入逻辑或清空逻辑。
- 为日志区新增单独的持久化配置项。
- 重新设计右侧详情栏的整体布局比例。

## Scope

主要改动：

- `src/atv_player/ui/player_window.py`

主要验证：

- `tests/test_player_window_ui.py`

## Design

### Layout Strategy

保留当前结构不变：

- `details`
- `metadata_section`
- `log_section`
- `log_view`

高度约束作用于 `log_section`，而不是只作用于 `log_view`。这样可以确保“日志整个区域”都被限制在 `1/4` 内，包含 `播放日志` 标题和日志文本框本身，避免只限制文本框后导致整个分区总高度超出目标。

### Height Rule

`log_section` 的 `maximumHeight` 应根据 `details` 当前高度动态计算：

- 目标上限 = `details` 当前高度的 `1/4`
- 使用整数像素值
- 当 `details` 高度变化时，重新应用该上限

这样窗口放大或缩小时，日志区上限会同步变化，但仍然始终保持相对比例约束。

### Update Trigger

需要在以下时机重新计算并应用日志区最大高度：

- 播放器窗口完成界面构建后
- 右侧详情栏高度发生变化时

实现上应保持为 `PlayerWindow` 内部的小型私有逻辑，不引入新的通用布局抽象。

### Behavior Compatibility

以下行为保持不变：

- `播放日志` 开关仍然控制 `log_section` 的显示与隐藏
- `详情` 开关仍然控制整个详情栏显示与隐藏
- `log_view` 继续只读并通过滚动查看历史日志
- fullscreen 和 wide mode 的现有侧栏显示规则不变

如果日志区被隐藏，最大高度约束仍可保留在控件上，但不需要改变任何现有可见性逻辑。

## Testing

补充 UI 测试覆盖：

- `log_section` 的最大高度会根据 `details` 当前高度更新为 `1/4`
- 当详情栏尺寸变化后，日志区高度上限会同步更新

不需要为本次改动新增持久化测试，因为该行为不引入新配置。

## Risks

- 如果高度计算绑定在错误的容器上，日志区比例可能相对整个窗口而不是右侧详情栏，导致结果偏差。
- 如果只限制 `log_view` 而不是 `log_section`，标题行会让整个日志区域超过 `1/4` 目标。
- 如果没有在尺寸变化时重算，窗口 resize 后日志区上限会失真。
