# 详细诊断导出设计

## 背景

当前 `F1` 帮助对话框已经提供：

- `一键复制`
- `导出诊断信息`

这两者现在都只输出简版系统信息，适合用户快速反馈版本、平台和组件环境。

当需要进一步排查问题时，简版信息不够用，但也不应该把现有简版导出继续做重，因为这会降低“快速复制/快速导出”的可用性。

## 目标

- 在现有 `F1` 帮助对话框中新增 `导出详细诊断` 按钮
- 保持现有 `一键复制` 和 `导出诊断信息` 行为不变
- 详细诊断导出为纯文本文件，便于直接阅读、贴聊天窗口或附到 issue
- 详细诊断首版只包含稳定、低风险、非敏感信息

## 非目标

- 不修改现有简版导出内容
- 不新增“简版/详细版”二选一弹窗
- 不把快捷键加入任何诊断导出内容
- 不导出 token、cookie、密码、认证头等敏感信息
- 不导出完整配置对象原文
- 不导出全量播放历史

## 方案

在帮助对话框现有按钮区新增第三个按钮：

- `一键复制`
- `导出诊断信息`
- `导出详细诊断`

按钮行为：

- `一键复制`：继续复制简版系统信息文本
- `导出诊断信息`：继续导出简版系统信息文本
- `导出详细诊断`：导出新的详细诊断文本文件

## 详细诊断文本结构

详细诊断使用固定纯文本结构，按以下顺序输出：

### 1. 系统信息

- `atv-player`
- `Python`
- `PySide6`
- `mpv`
- `ffmpeg`
- `yt-dlp`
- `Platform`

这部分直接复用现有系统信息采集结果，避免重复采集和文案漂移。

### 2. 运行环境

- Qt 平台插件名
- CPU 架构
- 可执行目录摘要
- 配置目录摘要
- 缓存目录摘要

这里的“路径摘要”只输出本地诊断有用的目录路径，不延伸到任意业务路径。

### 3. 应用配置摘要

首版只输出对排障有帮助且不敏感的配置摘要，例如：

- 主题
- 代理模式
- 后端地址
- 最后活动窗口
- 若已有明确布尔开关且与排障强相关，可按现有配置字段补充

不输出原始配置对象，也不输出敏感字段。

### 4. 插件摘要

- 已启用插件数量
- 已启用插件列表

如果列表较长，按一行一个输出，避免横向拥挤。

### 5. 最近日志

- 最近 `20` 条应用日志

日志保持现有展示文本风格即可，不额外做结构化编码。首版不做级别过滤，但保留时间顺序，确保错误现场仍可追踪。

## 文件名

详细诊断导出默认文件名：

`atv-player-diagnostics-detailed.txt`

简版继续保留：

`atv-player-diagnostics.txt`

## 实现边界

建议把“详细诊断文本生成”封装成单独构建函数，不塞进按钮点击逻辑里。帮助对话框只负责：

- 接收详细诊断文本
- 响应按钮点击
- 写文件

主窗口负责：

- 收集系统信息
- 组装详细诊断文本
- 传给帮助对话框

这样可以保持帮助对话框是纯 UI 层，避免它直接依赖配置、插件或日志源。

## 错误处理

- 用户取消保存：静默返回
- 写文件失败：沿用现有错误弹窗风格
- 某个详细字段采集失败：该字段写为 `不可用`，不应中断整个导出

## 测试

首版测试覆盖：

- 主窗口帮助对话框显示 `导出详细诊断` 按钮
- 点击 `导出详细诊断` 会写出目标文件
- 导出的默认文件名为 `atv-player-diagnostics-detailed.txt`
- 导出文本包含五段结构标题
- 简版复制和简版导出仍不包含快捷键
- 详细导出不包含快捷键
- 详细导出包含系统信息、运行环境、配置摘要、插件摘要、最近日志

## 兼容性与风险

- 本次只新增按钮和文本生成逻辑，不改变现有帮助入口和现有简版导出协议
- 主要风险是详细诊断文本采集范围过大导致耦合，因此首版严格限制为摘要信息
- 若后续要加入更多诊断字段，应继续遵循“非敏感、稳定、文本可读”的约束
