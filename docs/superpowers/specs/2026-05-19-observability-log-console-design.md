# Observability Log Console Design

## Summary

本次改造的目标是提升 atv-player 的可观察性和 troubleshooting 能力，覆盖两条日志链路：

- 播放窗口内的当前播放会话日志
- 应用后台的全局运行日志

改造后，播放器日志与后台日志会统一进入一套可持久化、可筛选、可导出的结构化日志体系；高级设置会新增 `日志` 标签页，用于查看、筛选、导出和清空日志。播放器窗口保留现有播放日志区，继续承担“当前播放上下文”的快速反馈职责。

## Goals

- 为播放链路补充关键节点日志，便于定位“点了什么、走到哪一步、失败在哪”
- 为后台运行、网络请求、代理、元数据、弹幕、插件、后台刷新等链路补充结构化日志
- 在应用内提供统一日志查看入口，而不是要求用户手动翻文件
- 让日志跨重启保留最近历史，并通过滚动压缩控制磁盘占用
- 支持按来源、级别、分类、关键字筛选日志
- 支持一键导出当前筛选结果和一键清空日志
- 支持用户关闭日志记录，以减少运行期开销

## Non-Goals

- 不做数据库化日志系统，不引入 SQLite 日志库
- 不做复杂的实时 tail、流式增量订阅或外部日志上传
- 不在播放器窗口直接展示完整 traceback
- 不把高频心跳类事件（如频繁进度上报）写入全局日志

## User Experience

### Player Window

- 保留当前播放窗口内的 `播放日志` 区域
- 继续以简洁中文消息为主，适合播放时即时观察
- 新增的关键日志点应覆盖：
  - 打开播放会话
  - 恢复播放 / 进入播放项
  - 解析播放地址开始 / 成功 / 失败
  - 线路、源组、清晰度、字幕、弹幕切换
  - HLS 代理准备、降级、回退
  - 自动切换线路
  - 停止播放
- 播放窗口日志保留现有时间戳格式和去重行为
- 播放窗口显示的日志消息要简洁，不直接暴露 traceback，只显示面向用户的失败摘要

### Advanced Settings

- `AdvancedSettingsDialog` 新增一个独立标签页 `日志`
- 用户进入该页即可查看日志，不再额外弹出“显示日志”窗口
- 页面布局分为三块：
  - 日志控制区：`启用日志记录` 开关
  - 顶部操作区：`导出日志`、`清空日志`、`刷新`
  - 筛选区：关键字、来源、级别、分类
  - 展示区：日志列表 + 详情面板
- 默认展示当前活跃日志和最近压缩归档，按时间倒序
- 默认只加载最近一批记录，避免设置页卡顿；首版上限为最近 `2000` 条
- `启用日志记录` 默认开启；关闭后停止新增日志写入，但仍可查看、导出、清空已有历史日志

### Filtering

- 关键字搜索范围：
  - `message`
  - `vod_name`
  - `episode_title`
- 来源筛选：
  - `全部`
  - `播放`
  - `后台`
- 级别筛选：
  - `全部`
  - `INFO`
  - `WARNING`
  - `ERROR`
- 分类筛选：
  - `全部`
  - `player`
  - `network`
  - `metadata`
  - `plugin`
  - `danmaku`
  - `live`
  - `app`

### Export

- 默认导出“当前筛选结果”，而不是导出原始全部文件
- 默认导出格式为纯文本 `.log`
- 导出的每条记录应展开为人类可读文本，至少包含：
  - 时间
  - 级别
  - 来源
  - 分类
  - 消息
  - 关键上下文（存在时）
- 首版不在 UI 中暴露 `.jsonl` / `.jsonl.gz` 原始格式导出选项

### Clear

- `清空日志` 会删除：
  - 当前活跃日志文件
  - 所有压缩归档日志文件
- 操作前弹确认框
- 清空后立即刷新列表，并显示空状态

### Logging Toggle

- 高级设置日志页顶部新增 `启用日志记录` 开关
- 该开关控制“未来是否继续记录日志”，不自动删除现有日志文件
- 关闭后：
  - 停止后台全局结构化日志写入
  - 停止播放器日志同步写入全局日志
  - 停止播放器窗口内详细播放日志的持续累积
- 关闭后仍保留必要的即时 UI 反馈，例如现有启动状态、错误提示、失败弹窗或播放失败状态展示；只是这些反馈不再进入日志系统
- 重新开启后仅对后续新日志生效，不补写关闭期间遗漏的日志

## Storage Design

### Location

- 日志存储目录：`app_data_dir() / "logs"`

建议文件布局：

- `application.jsonl`：当前活跃日志
- `application.YYYYMMDD-HHMMSS.jsonl.gz`：轮转后的压缩归档

### Record Format

使用结构化 `JSONL`，每行一条日志记录。基础字段如下：

- `timestamp`
- `level`
- `source`
- `category`
- `message`
- `module`

可选上下文字段如下：

- `vod_id`
- `vod_name`
- `episode_title`
- `session_id`
- `url_summary`
- `source_group_index`
- `source_index`
- `playlist_index`
- `proxy_mode`
- `exception`

另外新增配置项：

- `logging_enabled`

字段约束：

- `source` 取值限定为 `playback` 或 `app`
- `category` 取值优先使用固定枚举，避免首版出现大量零散值
- `url_summary` 只保存摘要，不直接保存可能过长的完整 URL
- `exception` 只用于全局结构化日志，不直接透传到播放器可见日志文本

## Retention And Rotation

- 活跃日志文件大小上限固定为 `10 MB`
- 超过上限后执行轮转：
  - 关闭当前活跃文件
  - 将旧文件压缩为 `.jsonl.gz`
  - 新建新的 `application.jsonl`
- 最多保留 `5` 个压缩归档文件
- 当压缩归档数量超过 `5` 个时，删除最旧归档
- 轮转、压缩、清理动作本身也写入全局日志，但要避免递归写爆；实现上应让日志轮转器内部使用最小、受控的记录路径

## Logging Pipeline

### Global App Logging

- 现有 `configure_logging()` 扩展为安装统一日志 handler，而不是只有 `basicConfig`
- 根 logger 继续输出到控制台/标准错误，便于开发阶段观察
- 同时新增一个写入结构化 `JSONL` 的 handler
- 后台代码继续使用 `logging.getLogger(__name__)`，但在关键路径补充更多上下文日志
- 当 `logging_enabled = false` 时，结构化日志 handler 直接短路，不执行序列化、写文件、轮转、压缩检查

### Playback Logging

- `PlayerWindow._append_log()` 在追加到窗口 `log_view` 的同时，也写入统一日志层
- 播放窗口写入全局日志时默认：
  - `source = "playback"`
  - `level = "INFO"`，失败路径按场景写 `WARNING` 或 `ERROR`
  - `category` 依据调用场景指定
- 若当前有活跃会话，则附带：
  - `vod_id`
  - `vod_name`
  - `episode_title`
  - `session_id`
  - 当前源组 / 线路 / 集数索引
- 当 `logging_enabled = false` 时，播放器详细日志追加应尽早返回，避免继续进行时间戳拼接、文本追加、结构化镜像写入

### Shared Logging Service

新增一层统一日志服务，职责包括：

- 接收结构化日志事件
- 负责序列化为 `JSONL`
- 负责文件大小检查、轮转、压缩、归档清理
- 提供日志读取接口，供高级设置日志页筛选展示
- 提供日志导出和清空接口

这层服务要避免直接和 Qt UI 绑定，便于在播放器窗口、后台控制器和设置页之间复用。

## Log Coverage

### Playback Category

播放器链路补充的关键日志点包括：

- 创建 / 打开播放会话
- 选择的源组、线路、播放列表索引
- 开始解析播放地址
- 解析完成并得到可加载地址
- 开始加载媒体
- 进入播放
- 播放失败
- 自动切换线路开始 / 成功 / 失败
- 手动切换清晰度、字幕、弹幕源
- 停止播放

每条日志尽量成对出现“开始 / 结果”，避免只打开始不打结果，或者同一动作刷出多条重复消息。

### Network Category

后台和播放准备链路补充：

- 是否命中代理
- 使用的代理模式
- 目标地址摘要
- 超时、403、连接失败、重试、降级
- HLS 上游请求、代理准备成功 / 失败

首版仍然遵守“少而关键”的原则，不为每一次普通成功请求都写一条详细日志。

### Metadata Category

- 元数据抓取开始 / 完成 / 失败
- 使用的 provider
- 命中的字段更新摘要
- 剧集标题增强开始 / 完成 / 失败

### Danmaku Category

- 弹幕搜索开始 / 完成 / 失败
- 预取开始 / 命中缓存 / 跳过
- 手动切换弹幕源
- 弹幕加载成功 / 失败

### Plugin Category

- 插件加载开始 / 完成 / 失败
- 插件刷新开始 / 完成 / 失败
- 关键异常信息写入全局日志

### Live And Background Tasks

- Live / EPG 后台刷新开始 / 完成 / 失败
- 自定义直播源刷新开始 / 完成 / 失败

### App Category

- 应用启动
- 登录成功 / 登录失败 / 登录过期回退
- 配置保存
- 日志轮转、压缩、归档清理
- 日志开关变更

## Advanced Settings Log Tab Design

### List View

展示列：

- 时间
- 级别
- 来源
- 分类
- 消息

列表默认按时间倒序显示。行内容优先让用户快速扫到“什么时候、哪个链路、报了什么”。

### Detail View

选中一条日志后展示详情，包括：

- 模块名
- `vod_id`
- `vod_name`
- `episode_title`
- `session_id`
- URL 摘要
- 异常摘要
- 完整消息

若某些字段为空，不显示无意义占位文本。

### Empty And Error States

- 没有日志时显示空状态文案
- 读取归档失败时，页面给出失败提示，同时保留已成功读取的部分
- 解析单条损坏日志时跳过该条，并在后台全局日志记录读取告警
- 当日志记录已关闭时，页面显示明确提示，例如“日志记录已关闭，当前仅可查看历史日志”

## Error Handling

- 日志写入失败不能影响主业务链路
- 结构化日志服务内部出错时：
  - 不抛异常到 UI
  - 尽量降级为静默失败或最小告警
- 压缩归档失败时：
  - 保留原始轮转文件
  - 记录全局错误日志
  - 不阻断后续新日志写入
- 日志读取失败时：
  - 高级设置日志页显示错误提示
  - 不影响其他设置项使用

## Performance Constraints

- 单条日志写入必须轻量，不做昂贵同步解析
- 播放进度上报等高频路径不进入全局日志
- 日志页默认只读取最近 `2000` 条，避免一次性把全部归档解压进内存
- 首版读取归档可接受串行扫描，但接口设计要留出未来优化空间
- 关闭日志后，日志相关路径应尽量在入口短路，避免无意义的字符串拼接、JSON 序列化和 UI 文本累积

## Testing Strategy

测试覆盖应至少包括以下部分：

### Logging Service

- 写入结构化 `JSONL`
- 活跃文件超过 `10 MB` 后轮转
- 旧文件压缩为 `.jsonl.gz`
- 压缩归档最多保留 `5` 个
- 超量时删除最旧归档
- 读取活跃日志和压缩归档
- 导出当前筛选结果为纯文本 `.log`
- 清空日志删除活跃文件和归档文件
- `logging_enabled = false` 时不写入新日志、不触发轮转

### Player Window

- `append_status_log()` / `_append_log()` 继续更新窗口日志
- 同步写入全局结构化日志
- 播放关键节点新增日志消息
- 失败路径写入正确级别和分类
- 不因为日志写入失败而中断播放 UI
- `logging_enabled = false` 时停止详细日志累积，但即时错误反馈仍保留

### Global Logging

- `configure_logging()` 安装全局结构化 handler
- 后台 logger 输出进入统一日志文件
- `exc_info` 异常被正确序列化为结构化字段
- `logging_enabled` 状态切换可持久化，并在重启后继续生效

### Advanced Settings UI

- 新增 `日志` 标签页
- 新增 `启用日志记录` 开关
- 默认加载最近日志
- 关键字、来源、级别、分类筛选生效
- 点击 `刷新` 重新加载
- 点击 `导出日志` 导出当前筛选结果
- 点击 `清空日志` 弹确认并清空
- 日志详情面板随选中项刷新

## Implementation Notes

- 保持播放器日志与全局日志职责分离：
  - 播放器日志负责即时反馈
  - 全局日志负责持久化与排障
- “关闭日志”针对的是详细日志记录，不等于移除必要的即时错误提示
- 首版采用结构化 `JSONL` 而不是 SQLite，是为了在复杂度与可维护性之间取平衡
- 高级设置日志页使用统一读取接口，不直接拼接文件读取逻辑到对话框内部
- 后台新增日志时优先补充上下文，而不是简单追加更多字符串

## Open Decisions Resolved

以下边界已经固定，不再在实现阶段重新发散：

- 日志查看范围：播放器日志与后台日志合并显示，并可按来源筛选
- 日志保留策略：跨重启保留最近历史
- 截断策略：按大小轮转，不按天数清理
- 大小与归档：活跃文件 `10 MB`，最多保留 `5` 个压缩归档
- 导出默认格式：纯文本 `.log`
- 日志记录可关闭，默认开启
