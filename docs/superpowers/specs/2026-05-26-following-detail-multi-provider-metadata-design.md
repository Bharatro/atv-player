# 追更详情页多平台元数据设计

## 概要

将“我的追更”详情页里的元数据展示，从当前单一快照升级为：

- 默认 `合并视图`
- 可切换的多平台原始视图
- 并列评分展示
- 独立的播放平台区

本轮的核心规则固定为：

- `TMDB` 作为主元数据来源
- `Bangumi / 豆瓣` 只补充缺失文本字段，不覆盖 `TMDB` 已有字段
- `TMDB / 豆瓣 / Bangumi` 评分并列显示，有值才显示
- `B站 / 爱奇艺 / 腾讯 / 优酷 / 芒果 / 搜狐` 负责播放平台链接、更新时间、最新集数、更新状态
- `TMDB` 的 watch provider 数据只用于“哪些平台可看”和平台链接候选，不直接充当更新状态真值来源
- 所有非 `TMDB` 平台都必须先经过独立匹配和置信度阈值过滤，未达阈值的平台直接不显示

这次改动保持现有追更详情页布局骨架，不把页面重做成 provider 管理界面，而是在现有详情体验上增加“多平台透明度”和“播放状态准确性”。

## 目标

- 让追更详情页默认显示一份稳定、可信的合并元数据结果。
- 让用户可以切换查看 `TMDB / 豆瓣 / Bangumi / 播放平台` 的原始元数据快照，便于核对和排错。
- 在顶部显式展示 `TMDB / 豆瓣 / Bangumi` 三平台评分。
- 将 `更新时间 / 最新集数 / 更新状态 / 播放链接` 从普通 metadata 文本列表中分离出来，形成独立播放平台区。
- 以 `TMDB` 条目为主身份，驱动 `Bangumi / 豆瓣 / B站 / 爱优腾 / 芒果 / 搜狐` 的独立匹配。
- 对于低置信度匹配的平台，宁可缺失也不误绑。

## 非目标

- 不新增用户可配置的 provider 排序界面。
- 不在本轮引入更多播放平台，第一版只覆盖 `B站 / 爱奇艺 / 腾讯 / 优酷 / 芒果 / 搜狐`。
- 不把追更详情页重做成多标签浏览器式页面。
- 不让 `Bangumi / 豆瓣` 反向覆盖 `TMDB` 的海报、背景或已存在结构化字段。
- 不把 `TMDB` 的 watch provider 数据直接当作“最新集数 / 更新时间”的真值来源。
- 不为低置信度平台显示“候选结果”或“疑似匹配”占位。

## 变更范围

主要涉及：

- `src/atv_player/following_metadata.py`
- `src/atv_player/following_models.py`
- `src/atv_player/controllers/following_controller.py`
- `src/atv_player/ui/following_detail_page.py`
- 相关 metadata provider / 匹配 / merge 层
- `tests/test_following_metadata.py`
- `tests/test_following_detail_page_ui.py`

可能补充涉及：

- `tests/test_metadata_matching.py`
- `tests/test_metadata_merge.py`
- 播放平台 provider 的测试文件

不涉及：

- 添加追更弹窗的 discovery 结构
- 主窗口搜索页
- 播放器详情页
- 追更记录的基础存储模型大改

## 现状问题

当前追更详情页虽然已经能显示一定的 metadata 信息，但整体仍是“单快照思路”：

- 页面只渲染一份 `FollowingDetailSnapshot`
- 很多字段最终被压平为一段文本
- 页面无法区分字段来自哪个 provider
- `更新时间 / 更新状态` 被显式过滤，不再展示
- 用户无法查看 `TMDB / 豆瓣 / Bangumi / 站点 provider` 的原始结果
- 评分无法稳定并列呈现多个平台

这带来的具体问题是：

- 页面默认结果不透明，用户难以判断某个字段来自哪里
- 一旦某个平台匹配偏了，用户无法自查
- “播放平台”与“数据库型元数据”混在一起时，更新状态来源不清楚
- 即使 `TMDB` 有 watch provider 数据，也缺少与站点 provider 的职责边界

## 设计目标

### 体验目标

- 默认看到的是一份高可信度、信息密度高的 `合并视图`
- 需要核对时，可以一键切到某个平台原始视图
- 播放平台信息独立成块，直接可点击打开链接
- 评分在顶部即可快速比较，不需要埋进详情文本

### 数据目标

- 将“最终合并结果”和“各 provider 原始结果”拆开建模
- 让每个平台独立匹配、独立阈值判定、独立纳入或剔除
- 明确不同 provider 在字段层面的职责边界

### 风险控制目标

- 匹配不到的平台不显示
- 某个平台失败不影响整体详情页
- 站点 provider 的更新数据和 `TMDB` 的链接数据不互相污染

## 总体方案

追更详情页新增一个“元数据来源区”，但默认仍以 `合并视图` 为主，不把页面主任务从“看简介、看进度、看分集”变成“研究 provider”。

最终交互结构：

1. 页面默认展示 `合并` 视图
2. 顶部新增评分条，固定顺序显示：
   - `TMDB`
   - `豆瓣`
   - `Bangumi`
3. 元数据区顶部新增来源切换：
   - 固定 `合并`
   - 命中且过阈值的 `TMDB / 豆瓣 / Bangumi / 播放平台`
4. `合并` 视图显示最终用户应该消费的统一结果
5. 切到某个平台时，正文显示该 provider 的原始快照
6. 播放平台信息在 `合并` 视图中独立成块，不再混入普通 detail fields

## 页面结构

### 默认合并视图

默认视图继续沿用当前详情页的主体布局，但元数据展示区拆成三层：

1. 摘要层
   - 标题
   - 追更进度
   - 最新集数
   - 总集数
2. 评分层
   - `TMDB`
   - `豆瓣`
   - `Bangumi`
3. 元数据内容层
   - 普通概览字段
   - 播放平台区
   - 简介

### 来源切换

来源切换放在元数据内容区顶部，建议表现为一组轻量 tab / segmented buttons。

显示规则：

- `合并` 永远显示
- `TMDB` 只要当前详情有明确 `TMDB` 身份即显示
- `豆瓣 / Bangumi / B站 / 爱奇艺 / 腾讯 / 优酷 / 芒果 / 搜狐` 只有在：
  - 匹配成功
  - 置信度达到阈值
  - 成功拿到可展示快照
  时才显示

### 播放平台区

`播放平台区` 是 `合并` 视图里的独立块，不再作为普通文本字段的一部分。

每个平台一行，内容包括：

- 平台名
- 平台链接
- 最新集数
- 更新时间文本
- 更新状态文案

显示原则：

- 如果只有 `TMDB` 的平台链接，没有站点 provider 更新信息，则只显示平台名和链接，不显示伪造的更新时间或更新状态
- 如果站点 provider 提供了更新信息，则将其补到该平台行

## 数据模型

当前详情页返回的是单一 `FollowingDetailView(record, snapshot)` 结构。本轮需要升级成“合并结果 + provider 原始结果”的 bundle。

### `FollowingMetadataSourceSnapshot`

用于表示单个平台的原始快照。建议字段：

- `provider`
- `provider_label`
- `provider_id`
- `matched`
- `confidence`
- `url`
- `rating`
- `overview`
- `detail_fields`
- `episodes`
- `seasons`
- `playback_platform_entries`

约束：

- 未达到阈值的平台不生成这个对象
- 原始视图尽量保留 provider 自己的原始信息表达，不做字段级混合

### `PlaybackPlatformEntry`

用于统一承接播放平台信息。建议字段：

- `provider`
- `label`
- `url`
- `latest_episode`
- `update_time_text`
- `status_text`

说明：

- `url` 可以来自 `TMDB` watch provider 或站点 provider
- `latest_episode / update_time_text / status_text` 优先来自站点 provider

### `FollowingMetadataBundle`

用于挂在详情 view 上，表示当前详情页所有 metadata 视图数据。建议字段：

- `merged_snapshot`
- `source_snapshots`
- `available_source_keys`
- `default_source_key`

其中：

- `merged_snapshot` 是默认渲染对象
- `source_snapshots` 存储 `TMDB / 豆瓣 / Bangumi / 各播放平台` 的原始快照
- `default_source_key` 固定为 `merged`

## 字段优先级

### 视觉主字段

由 `TMDB` 提供：

- 海报
- 背景图
- 年份
- 类型
- 季结构骨架
- 分集 still

这些字段不接受 `豆瓣 / Bangumi / 站点 provider` 在合并视图中反向覆盖。

### 评分字段

评分不走覆盖逻辑，而是并列显示。

固定顺序：

1. `TMDB`
2. `豆瓣`
3. `Bangumi`

规则：

- 有值才显示
- 没值就不占位
- 不因为某个平台评分更高就覆盖另一个平台

### 文本补充字段

`豆瓣 / Bangumi` 只补充 `TMDB` 缺失值，例如：

- 中文简介
- 中文演员 / 导演文本
- 别名
- 动漫向文本信息

规则：

- `TMDB` 已有值时，不允许 `豆瓣 / Bangumi` 覆盖
- `Bangumi` 的分集标题和分集简介如果更完整，第一版仅用于该 provider 原始视图；合并视图仍以 `TMDB` 季集结构为主

### 播放状态字段

以下字段统一由播放平台区承接，不再混进普通 metadata 文本：

- `更新时间`
- `最新集数`
- `更新状态`
- `播放链接`

来源规则：

- `TMDB`：提供平台存在性和平台链接候选
- 站点 provider：提供 `最新集数 / 更新时间 / 更新状态`

## 匹配策略

所有非 `TMDB` 平台都以当前已绑定的 `TMDB` 条目为主身份，不再让平台之间相互串搜。

### 查询基准

每个平台匹配时，都以 `TMDB` 主信息为基准组合查询：

- 标题
- 原始标题
- 年份
- 媒体类型
- 季号
- 别名

### 匹配维度

每个平台独立计算置信度，建议维度包括：

- 标题命中
- 别名命中
- 年份一致或接近
- 媒体类型一致
- 季号一致
- 总集数接近

对播放平台再增加额外加权：

- 是否存在明确 `season / album` 身份
- 平台返回的最新集数与 `TMDB` 季结构是否合理接近

### 候选选择和阈值

每个平台只取最高分候选。

纳入规则：

- 最高分候选达到该平台阈值：纳入 bundle
- 最高分候选未达到阈值：该平台直接不显示

页面上不提供：

- 低置信度占位
- “可能是”候选
- 用户手动二选一确认

## 各平台补充内容范围

### `Bangumi`

在第一版里补充：

- 评分
- 中文 / 日文别名
- 动漫向简介补充
- 动漫特有文本信息

原始视图中可以保留：

- 单集标题
- 单集简介

### `豆瓣`

在第一版里补充：

- 评分
- 中文简介
- 中文演员 / 导演文本
- 别名

### `B站 / 爱奇艺 / 腾讯 / 优酷 / 芒果 / 搜狐`

在第一版里补充：

- 平台链接
- 平台标题
- 最新集数
- 更新时间文本
- 更新状态文案
- 可播放 / 在更状态

平台侧封面、平台侧简介等如果存在，第一版只在该平台原始视图里展示，不进入合并视图的主视觉或主文本覆盖链。

## 数据流

### 详情加载

打开追更详情页时：

1. 先加载基础 `FollowingRecord + FollowingDetailSnapshot`
2. 确认当前条目拥有明确 `TMDB` 身份
3. 以 `TMDB` 为主身份驱动：
   - `Bangumi`
   - `豆瓣`
   - `B站 / 爱奇艺 / 腾讯 / 优酷 / 芒果 / 搜狐`
4. 每个平台独立搜索、独立评分、独立阈值判定
5. 构造 `source_snapshots`
6. 基于字段优先级构造 `merged_snapshot`
7. 渲染默认 `合并` 视图

### 播放平台区组装

播放平台区的组装流程：

1. 从 `TMDB` watch provider 数据中提取：
   - 平台存在性
   - 平台链接
2. 从站点 provider 中提取：
   - 最新集数
   - 更新时间文本
   - 更新状态文案
3. 按平台合并到 `PlaybackPlatformEntry`
4. 没有站点更新信息的平台，只保留链接展示

### 视图切换

详情页内部保存当前选中的 `source_key`：

- `merged`
- `tmdb`
- `douban`
- `bangumi`
- `bilibili`
- `iqiyi`
- `tencent`
- `youku`
- `mgtv`
- `sohu`

切换时只重绘元数据展示区，不重建整页。

## UI 细节

### 评分条

评分条放在详情页顶部摘要区附近，显示格式例如：

- `TMDB 8.1`
- `豆瓣 7.9`
- `Bangumi 8.4`

规则：

- 固定顺序显示
- 只显示有值的平台
- 不补空占位

### 原始视图

切到某个平台原始视图时：

- 页面正文展示该 provider 的原始简介
- 展示该 provider 的 detail fields
- 如果该 provider 有季 / 分集原始信息，可展示其原始数据
- 页面顶部标题、主海报和整体详情页框架保持统一，不需要整页跳转

### 合并视图中的详情文本

现有诸如：

- 类型
- 年代
- 地区
- 语言
- 导演
- 演员
- 别名
- 简介

仍然以文本块方式展示，但填充值从“单快照来源”改成“按字段优先级合并后的结果”。

## 容错与失败处理

- 某个平台搜索失败，不影响其他平台和主页面渲染
- 某个平台详情失败，不影响 `merged` 视图已有内容
- 某个平台置信度不足，直接不显示
- 平台链接存在但更新信息缺失时，不显示虚假更新时间或更新状态
- `TMDB` 缺字段时，才允许 `豆瓣 / Bangumi` 填空
- `更新时间 / 更新状态` 不再作为普通 detail field 混回概览文本

## 测试

### 数据层测试

应覆盖：

- `TMDB` 为主字段来源，`豆瓣 / Bangumi` 只补缺失，不覆盖已有值
- 评分条数据按 `TMDB / 豆瓣 / Bangumi` 顺序产出
- 未达到阈值的平台不会出现在 `source_snapshots`
- `TMDB` watch provider 只提供链接 / 平台存在性，不直接产出更新状态真值
- 站点 provider 可以为对应平台补上 `最新集数 / 更新时间 / 更新状态`
- 若某平台只有链接无更新信息，bundle 中仍保留链接，但相关状态字段为空

### UI 测试

应覆盖：

- 默认打开详情页时显示 `合并` 视图
- 页面显示评分条，且顺序固定为 `TMDB / 豆瓣 / Bangumi`
- 来源切换只显示命中且过阈值的平台
- 切换 `合并 / provider 原始视图` 时，元数据内容区正确刷新
- `播放平台区` 独立显示，不再混入 overview 文本
- 现有被跳过的 `更新时间 / 更新状态` 字段迁移到播放平台区后仍可见
- 点击播放平台链接时触发浏览器打开动作

## 风险

- 如果 bundle 设计不清晰，容易在现有单快照结构上继续打补丁，导致 UI 和数据层再度耦合
- 播放平台 provider 的更新数据质量可能不完全一致，需要通过阈值和空值策略维持稳定体验
- 原始视图和合并视图共存后，测试很容易写成过度绑定具体 widget 结构
- 若 `TMDB` 身份缺失或不稳定，需要为没有 `TMDB` 的追更保留旧路径或安全降级

## 验收标准

- 追更详情页默认显示 `合并` 视图
- 页面顶部可并列显示 `TMDB / 豆瓣 / Bangumi` 评分，有值才显示
- 页面提供来源切换，可查看命中平台的原始元数据视图
- 合并视图中，`TMDB` 负责主结构化字段，`豆瓣 / Bangumi` 只补缺失文本
- `B站 / 爱奇艺 / 腾讯 / 优酷 / 芒果 / 搜狐` 可在播放平台区显示链接和更新信息
- `TMDB` watch provider 仅用于平台存在性和链接候选，不直接承担更新状态真值
- 未达到阈值的平台不会显示
- `更新时间 / 最新集数 / 更新状态` 不再混进普通元数据文本，而是进入独立播放平台区
