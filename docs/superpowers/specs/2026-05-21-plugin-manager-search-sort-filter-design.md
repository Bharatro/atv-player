# Plugin Manager Search Sort Filter Design

## Summary

当前插件管理对话框会直接把 `plugin_manager.list_plugins()` 的完整结果渲染到表格里。随着插件数量增多，用户只能依赖肉眼查找，无法快速定位某个插件，也无法临时按名称或最近加载时间重排视图。

本次改动目标是在不改变插件底层持久顺序语义的前提下，为插件管理对话框补上一组轻量、常驻、即时生效的列表工具：搜索、启用状态筛选、视图层排序。

## Goals

- 在插件管理对话框中提供常驻搜索、筛选、排序入口。
- 搜索支持按插件名称和地址即时过滤。
- 筛选支持 `全部`、`仅启用`、`仅禁用` 三种状态。
- 排序支持 `当前顺序`、`名称`、`最近加载` 三种视图排序方式。
- 所有条件变化都即时刷新表格，不需要额外点击“应用”。
- 保持现有插件真实顺序模型不变，避免和持久排序入口冲突。

## Non-Goals

- 不增加新的持久排序语义。
- 不改变 `上移`、`下移`、`调整顺序` 的真实排序职责。
- 不新增来源筛选、错误状态筛选或组合排序。
- 不修改 `plugin_manager`、repository 或插件数据模型。
- 不把搜索、筛选条件持久化到应用重启之后。

## Current Flow

当前插件管理 UI 位于 [plugin_manager_dialog.py](/home/harold/workspace/atv-player/src/atv_player/ui/plugin_manager_dialog.py:33)。

- `reload_plugins()` 直接读取 manager 返回的完整插件列表并填充表格。
- 表格没有搜索、筛选、排序工具。
- `上移` / `下移` 直接依赖当前表格行位置判断是否可用。
- `调整顺序` 会打开专用排序对话框，负责修改真实 `sort_order`。

这意味着当前主对话框同时缺少“快速定位”和“临时视图整理”能力。用户如果只想找一个插件，只能滚动查找；如果只想临时按名称浏览，也会和真实顺序操作混在一起。

## Proposed Behavior

### Condition Bar Layout

在警告文案下方、现有按钮行上方新增一行常驻条件栏，从左到右包含：

- 搜索框
- 启用筛选下拉框
- 排序方式下拉框
- `清空` 按钮

搜索框占主要宽度，其余控件保持紧凑。表格结构仍然保留现有 7 列，不新增展示列。

搜索框占位文案使用 `搜索名称或地址`。

### Search Behavior

搜索仅匹配两个字段：

- `display_name`
- `source_value`

匹配规则：

- 去掉用户输入首尾空白后再匹配
- 使用不区分大小写的包含匹配
- 搜索词为空时，不施加搜索过滤

搜索输入变化后立即更新表格。

### Filter Behavior

启用筛选下拉框提供：

- `全部`
- `仅启用`
- `仅禁用`

切换后立即更新表格。

筛选只基于 `enabled` 布尔值判断，不读取其他状态字段。

### Sort Behavior

排序方式下拉框提供：

- `当前顺序`
- `名称`
- `最近加载`

语义如下：

- `当前顺序`
  - 按插件真实 `sort_order` 显示
  - 这是默认值
- `名称`
  - 按显示名称升序
  - 若显示名称为空，则回退到地址文本参与排序
  - 同值时使用原始 `sort_order` 作为稳定的并列打破规则
- `最近加载`
  - 按 `last_loaded_at` 降序
  - 有加载时间的排前面，没有加载时间的排最后
  - 同值时使用原始 `sort_order` 作为稳定的并列打破规则

排序方式切换后立即更新表格。

### Clear Behavior

`清空` 按钮一键恢复默认条件：

- 搜索词清空
- 启用筛选回到 `全部`
- 排序方式回到 `当前顺序`

恢复后立即刷新表格。

## Selection And Action Semantics

### View-Only Ordering

新的排序只作用于当前表格视图，不会写回真实插件顺序。

真实顺序仍然只通过现有入口修改：

- `上移`
- `下移`
- `调整顺序`

这保证“临时看法”和“持久顺序”不混用。

### Action Targeting

无论用户当前是否经过搜索、筛选或视图排序，表格中的每一行仍然绑定真实 `plugin_id`。因此下列动作继续作用于真实插件对象：

- 编辑名称
- 编辑配置
- 分类管理
- 启用/禁用
- 刷新
- 查看日志
- 删除

过滤和排序只改变当前可见行集合与显示顺序，不改变动作目标。

### Move Button Behavior

`上移` 和 `下移` 依赖真实顺序语义，因此只在排序方式为 `当前顺序` 时可用。

当排序方式切换到 `名称` 或 `最近加载` 时：

- `上移` 禁用
- `下移` 禁用

`调整顺序` 按钮继续保持可用，因为它是显式进入持久排序对话框，而不是依赖当前表格视图直接改序。

### Selection Preservation

条件变化或调用 `reload_plugins()` 后，优先按 `plugin_id` 恢复之前的单选项：

- 如果该插件仍在当前可见结果中，则恢复选中
- 如果该插件已被过滤掉，则清空选择

多选状态在条件变化后不强求完整恢复。新的条件应用后，允许退回到无选择状态，以避免恢复逻辑和当前可见行不一致。

## Empty State And Refresh Behavior

### Empty Results

当当前条件下没有任何匹配插件时：

- 表格显示 0 行
- 不弹出提示框
- 所有依赖选择的按钮保持禁用
- 对话框显示一条轻量提示文案，例如 `没有匹配的插件`

用户可通过修改条件或点击 `清空` 回到默认视图。

### Reload Semantics

`reload_plugins()` 仍然先获取完整插件列表，然后再在对话框内部应用当前搜索、筛选、排序条件，最终渲染当前可见视图。

这意味着：

- 添加插件后，当前条件仍然保留
- 导入插件后，当前条件仍然保留
- 改名、启用/禁用、刷新、删除后，当前条件仍然保留

用户完成任意管理动作后，不会被强制弹回默认列表状态。

## Architecture

### UI-Only State

实现只落在 [plugin_manager_dialog.py](/home/harold/workspace/atv-player/src/atv_player/ui/plugin_manager_dialog.py:33) 中，在对话框内部维护轻量视图状态：

- 当前搜索词
- 当前启用筛选值
- 当前排序方式

这些状态属于纯 UI 视图状态，不进入 manager、repository 或模型层。

### Rendering Pipeline

建议把现有 `reload_plugins()` 拆成两个逻辑阶段：

1. 获取完整插件列表并更新现有快照/脏状态判断
2. 基于当前条件计算当前可见插件列表并填充表格

为保证逻辑集中，过滤和排序可提取为独立私有方法，例如：

- `_visible_plugins(...)`
- `_matches_search(...)`
- `_sort_plugins(...)`

具体方法名可以调整，但需要保持“数据获取”和“视图计算”职责分离，避免 `reload_plugins()` 继续膨胀。

### Stable Row Identity

表格行仍然通过 `QTableWidgetItem(...).setData(256, plugin.id)` 保存真实插件 ID。所有选择恢复、动作执行和按钮状态计算都基于这个稳定 ID，而不是基于过滤后的行号推断真实插件对象。

## Error Handling

- 搜索、筛选、排序本身不引入新的错误提示路径。
- 如果某个已有管理动作失败，仍然沿用当前对话框已有的错误处理方式。
- 条件变化不会触发新的后台请求，因此不需要额外异步错误分支。

## Testing

测试集中补到 [test_plugin_manager_dialog.py](/home/harold/workspace/atv-player/tests/test_plugin_manager_dialog.py:1)。

至少覆盖以下场景：

- 对话框渲染出搜索框、启用筛选下拉框、排序下拉框、`清空` 按钮。
- 输入名称关键字时，只显示名称匹配的插件。
- 输入地址关键字时，只显示地址匹配的插件。
- 搜索大小写不敏感，并忽略首尾空白。
- `仅启用` 和 `仅禁用` 能正确过滤。
- `名称` 排序只改变当前表格显示顺序，不修改底层插件列表顺序。
- `最近加载` 排序把有时间的插件排前面、无时间的插件排后面。
- 当排序方式不是 `当前顺序` 时，`上移` / `下移` 被禁用。
- 点击 `清空` 会恢复默认条件和默认显示顺序。
- 在当前条件下调用 `reload_plugins()` 后，条件仍被保留并继续生效。
- 单选插件在重新渲染后如果仍可见，则按 `plugin_id` 恢复选中；否则清空选择。
- 过滤结果为空时，表格为 0 行且相关选择型按钮禁用。

## Risks

- 如果把视图排序错误地写回 `sort_order`，会和现有持久排序入口冲突。
- 如果 `上移` / `下移` 在视图排序模式下仍保持可用，用户会误以为它操作的是当前可见顺序，造成语义混乱。
- 如果 `reload_plugins()` 不保留当前条件，用户每做一个动作都会被打回默认列表，搜索和筛选价值会明显下降。

## Recommendation

采用“常驻条件栏 + 即时视图过滤/排序 + 持久顺序入口维持现状”的方案。

这是对现有插件管理对话框最稳妥的增强：

- 交互成本低，用户几乎不需要学习
- 不侵入 manager 和 repository
- 不改变真实排序模型
- 能显著改善插件数量增多后的查找效率
