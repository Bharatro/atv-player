# Plugin Manager Bulk Selection Actions Design

## Summary

在插件管理对话框中，插件表格已经支持多行选择，但当前只有 `删除` 真正支持多选，其它常用管理动作仍被限制为单选。这会导致用户想同时处理一批插件时，只能重复逐条点击。

本次改动目标是在保持现有插件数据模型不变的前提下，为插件管理对话框补齐多选批量操作能力，优先覆盖高频管理动作：启用、禁用、刷新、删除。

## Goals

- 保留现有整行多选交互，不新增复选列。
- 让插件管理支持对多选结果批量执行 `启用`、`禁用`、`刷新`、`删除`。
- 用明确语义按钮替代当前单个 `启用/禁用` 切换按钮，避免混合选中时行为歧义。
- 批量操作后保留当前搜索、筛选、排序条件。
- 不修改插件真实顺序语义，不影响现有单选专属动作。

## Non-Goals

- 不新增 repository 或 manager 级批量 API。
- 不新增表格复选框列、工具栏计数器或额外批量确认面板。
- 不改变 `编辑名称`、`编辑配置`、`分类管理`、`查看日志` 的单选语义。
- 不让插件自定义动作支持多选执行。
- 不调整 `上移`、`下移`、`调整顺序` 的职责。

## Current Flow

当前实现位于 [plugin_manager_dialog.py](/home/harold/workspace/atv-player/src/atv_player/ui/plugin_manager_dialog.py:33)。

- 表格启用了 `ExtendedSelection`，可以多选整行。
- 按钮状态由 `_sync_action_state()` 统一控制。
- 只有 `删除` 在多选时仍然可用。
- `启用/禁用` 目前是单个切换按钮，只能对当前单选项执行“反转 enabled 状态”。
- `刷新` 也只能单选。

这意味着底层交互能力和上层动作语义不一致：UI 允许多选，但大多数实际管理动作并没有跟上。

## Proposed Behavior

### Selection Model

继续使用现有表格的整行多选：

- 不新增第一列复选框
- 不改成复选卡片式列表
- 不改变当前选择模型和快捷键行为

这样可以复用现有测试辅助和选择恢复逻辑，避免引入第二套选中状态源。

### Action Layout

把当前按钮行中的 `启用/禁用` 拆成两个明确动作：

- `启用`
- `禁用`

批量能力矩阵如下：

- `编辑名称`：仅单选
- `编辑配置`：仅单选
- `分类管理`：仅单选
- `启用`：支持多选
- `禁用`：支持多选
- `上移`：仅单选
- `下移`：仅单选
- `调整顺序`：不依赖选择
- `刷新`：支持多选
- `查看日志`：仅单选
- `删除`：支持多选

### Enable / Disable Semantics

批量启用和禁用都采用“只处理需要变更的项”语义，而不是对每个选中项做状态反转。

`启用`：

- 对选中集合中 `enabled=False` 的插件逐个调用启用
- 已经启用的项直接跳过

`禁用`：

- 对选中集合中 `enabled=True` 的插件逐个调用禁用
- 已经禁用的项直接跳过

这样在混合选中场景下语义稳定且可预测。例如选中 10 个插件，其中 6 个已启用、4 个已禁用，点击 `启用` 后只有那 4 个禁用插件被修改。

### Refresh Semantics

`刷新` 只要存在选择就可用。

执行时：

- 按当前表格选中行对应的 `plugin_id` 顺序逐个调用 `refresh_plugin`
- 不因为某个插件已启用或禁用而跳过

本次不新增并发刷新，也不为单条失败建立新的局部汇总面板，继续沿用当前逐次调用和既有错误处理方式。

### Delete Semantics

`删除` 继续支持多选，但要和新的批量语义保持一致：

- 按当前选中行对应的 `plugin_id` 顺序逐个删除
- 删除完成后统一刷新列表一次

如果现有删除流程带确认框，则仍然沿用一份确认，不为每条记录重复弹窗。

## Button State Rules

按钮启用规则从“是否单选”扩展为“是否存在可执行目标”。

### Multi-Selection Buttons

`启用` 可用条件：

- 当前至少选中一项
- 且选中集合里至少有一个 `enabled=False`

`禁用` 可用条件：

- 当前至少选中一项
- 且选中集合里至少有一个 `enabled=True`

`刷新` 可用条件：

- 当前至少选中一项

`删除` 可用条件：

- 当前至少选中一项

### Single-Selection Buttons

以下按钮继续严格要求单选：

- `编辑名称`
- `编辑配置`
- `分类管理`
- `查看日志`
- `上移`
- `下移`

当选择数量不是 1 时，它们保持禁用。

### Reorder Buttons

`上移`、`下移` 仍然同时受两个条件约束：

- 必须单选
- 当前排序方式必须是 `当前顺序`

这部分行为与现有搜索/排序视图语义保持一致。

## Selection Preservation And View Behavior

### Reload Strategy

批量动作执行完成后统一调用一次 `reload_plugins()`，而不是每处理一个插件就重载一次 UI。

这样可以：

- 减少重复重绘
- 保持按钮状态更新路径集中
- 保持现有搜索、筛选、排序条件不丢失

### Selection Restoration

当前 `reload_plugins()` 只优先恢复单选项，本次需要扩展为按 `plugin_id` 恢复仍然可见的选中集合。

规则如下：

- 批量启用 / 禁用 / 刷新后，优先恢复操作前仍然存在且仍然可见的选中插件
- 批量删除后，被删插件自然不恢复；未删且仍可见的选中插件继续恢复
- 如果因为当前筛选条件导致某些项操作后不再可见，则不强行恢复这些项

例如在 `仅禁用` 过滤下批量启用几项插件，操作完成后这些插件会从当前视图中消失，这是预期行为，不需要额外提示。

### Custom Plugin Action Area

插件自定义动作区域继续只服务于单个插件：

- 单选时按当前插件加载动作
- 无选择时显示 `无动作`
- 多选时也显示 `无动作`

这样可以避免把插件自定义动作错误地理解为宿主批量动作。

## Architecture

实现范围限制在 [plugin_manager_dialog.py](/home/harold/workspace/atv-player/src/atv_player/ui/plugin_manager_dialog.py:33) 和对话框测试中。

### UI-Only Batch Dispatch

批量操作先不下沉到 `PluginActions` 或 `SpiderPluginManager` 新增公共批量接口，而是在对话框中利用现有单条 API 逐个分发：

- 启用 / 禁用：复用 `plugin_actions.apply_toggle_enabled(...)` 或直接调用现有 manager 启停入口的等价路径
- 刷新：复用现有 `refresh_plugin(...)`
- 删除：复用现有 `delete_plugin(...)`

关键点是：对话框先基于选中插件和当前状态筛出“需要执行的目标集合”，再统一执行，最后只重载一次。

### Stable Identity

所有批量目标仍然依赖表格第 0 列绑定的真实 `plugin_id`，不依赖显示名称或过滤后的行号以外的推断。

这保证：

- 排序变化不会影响动作目标
- 搜索 / 筛选后的视图仍能准确命中真实插件
- 批量恢复选择时可以继续按稳定 ID 工作

## Error Handling

- 不新增新的后台任务模型。
- 如果批量过程中某条操作抛错，继续沿用现有对话框错误提示方式。
- 本次不设计复杂的“部分成功摘要”面板。
- 如果实现中发现现有单条错误提示会导致批量操作中断，则以“保持行为一致、实现最小变更”为优先，只做当前功能所需的最小封装。

## Testing

测试主要补到 [test_plugin_manager_dialog.py](/home/harold/workspace/atv-player/tests/test_plugin_manager_dialog.py:1)。

至少覆盖：

- 对话框暴露 `启用` 和 `禁用` 两个明确按钮，不再依赖单个 `启用/禁用` 按钮。
- 多选时，单选专属按钮保持禁用。
- 多选集合中存在禁用项时，`启用` 可用；否则禁用。

- 批量禁用只处理当前启用项。
- 批量刷新按选中插件 ID 顺序调用。
- 批量删除按选中插件 ID 顺序调用。
- 批量操作后当前搜索 / 筛选 / 排序条件仍保留。
- 多选状态下插件自定义动作区显示 `无动作`。

## Result

改动完成后，插件管理对话框的多选能力将从“只能批量删除”扩展到“可以批量启用、禁用、刷新、删除”。用户可以在当前搜索、筛选、排序视图内直接处理一组插件，而不需要重复逐条点击，同时现有单选编辑类动作和真实排序语义保持不变。
