# 弹幕源对话框清除弹幕按钮设计

## 概要

在播放器的“弹幕源”对话框按钮区新增一个“清除弹幕”按钮，用于关闭当前播放项已经加载到播放器中的弹幕，同时保留已搜索到的候选弹幕源和当前选中来源，方便用户之后重新加载。

## 目标

- 在“弹幕源”对话框中提供显式的“清除弹幕”入口。
- 点击后立即移除当前播放器里正在显示的弹幕轨道。
- 清空当前 `PlayItem` 的 `danmaku_xml`，让当前播放项回到“无已加载弹幕”的状态。
- 保留候选源列表和当前选中的来源，用户可以直接再次点击“加载弹幕”。

## 非目标

- 不清空弹幕搜索结果缓存。
- 不清空当前播放项的 `danmaku_candidates`。
- 不清空 `selected_danmaku_url`、`selected_danmaku_provider`、`selected_danmaku_title`。
- 不新增或修改弹幕 controller API。
- 不调整已有“重新搜索”“恢复默认”“加载弹幕”的语义。

## 用户行为

当用户打开“弹幕源”对话框且当前播放项已经加载弹幕时：

- 对话框底部显示可点击的“清除弹幕”按钮。
- 点击后，当前视频上的弹幕立即消失。
- 候选源列表保持不变。
- 当前选中的来源保持不变。
- 用户后续仍可直接点击“加载弹幕”重新启用该来源。

当当前播放项没有已加载弹幕时：

- “清除弹幕”按钮显示为禁用状态。

当弹幕源后台任务正在运行时：

- “清除弹幕”按钮禁用，避免和搜索或切换流程并发修改同一播放项状态。

## 设计

### UI

在 `PlayerWindow._ensure_danmaku_source_dialog()` 的按钮区增加：

- `清除弹幕`

按钮顺序调整为：

- `重新搜索`
- `恢复默认`
- `清除弹幕`
- `加载弹幕`

### 状态更新

沿用现有 `PlayerWindow._refresh_danmaku_source_dialog_actions()` 统一刷新按钮可用状态。

“清除弹幕”按钮启用条件：

- 当前存在播放项。
- 当前播放项没有活跃的弹幕源后台任务。
- 当前播放项的 `danmaku_xml` 非空。

### 点击行为

新增 `PlayerWindow._clear_current_item_danmaku_source()`，执行以下步骤：

1. 读取当前播放项，若不存在则直接返回。
2. 调用现有 `_clear_active_danmaku()`，移除播放器里已经加载的弹幕轨道并清理临时文件。
3. 将当前播放项的 `danmaku_xml` 置空字符串。
4. 保留当前播放项的候选源列表、已选来源字段和搜索字段不变。
5. 调用 `_refresh_danmaku_source_dialog_actions(current_item)` 刷新按钮状态。

这里不需要更新状态文案，因为这是本地即时 UI 操作，不是异步搜索或下载任务。

## 架构取舍

推荐只在 `PlayerWindow` 内实现，不新增 controller 层接口。

原因：

- “清除弹幕”本质是当前播放器窗口的本地显示状态清理。
- 现有 `_clear_active_danmaku()` 已经完整封装了轨道移除与本地清理逻辑。
- 如果把这个动作下沉到 controller，会把纯 UI 的播放态控制扩散到数据/搜索层，收益很低。

## 风险与兼容性

- 需要确保清除后不会误清当前来源选择，否则会破坏“重新加载同一来源”的预期。
- 需要确保只清空 `danmaku_xml`，不影响后续自动/手动搜索逻辑。
- 需要确保按钮状态和现有异步任务状态联动，避免搜索中点击导致 UI 状态错乱。

## 验证

需要覆盖的 UI 回归测试：

- 打开“弹幕源”对话框时存在“清除弹幕”按钮。
- 当前播放项已有弹幕时，按钮为可点击。
- 点击后当前 `PlayItem.danmaku_xml` 被清空。
- 点击后播放器收到移除当前弹幕轨道的调用。
- 点击后候选源与当前已选来源保持不变。
- 清除完成后按钮变为禁用，但“加载弹幕”仍可继续使用。
