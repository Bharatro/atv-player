# 首次播放优化设计

**日期**: 2026-05-13

## 背景

当前播放器已经具备：

- 多来源播放列表与线路切换
- 内置解析器
- HLS 代理
- 播放恢复
- 字幕、弹幕、清晰度切换
- 播放日志

但“点击播放到首帧出现”的链路仍然偏重。对用户来说，最明显的问题不是单一功能缺失，而是首次播放等待时间长、加载阶段不透明、失败后恢复路径不够短。

本设计聚焦“用户体验 > 播放更顺 > 首次播放更快”，优先优化首次播放体感，不重做播放器整体架构。

## 目标

本轮设计目标：

1. 用户点击播放后，播放器明确显示当前所处阶段。
2. 对可缓存的解析结果做短时缓存，减少重复等待。
3. 让视频尽快起播，非关键附加信息允许稍后补齐。
4. 播放失败时，直接给出最短恢复路径。

非目标：

- 不引入全新播放状态机架构
- 不做多设备同步
- 不做下载或离线媒体库
- 不先做复杂的来源健康度学习
- 不在第一阶段做全自动多级回退

## 用户问题

当前体验中的主要痛点：

- 用户看不到播放器卡在哪个阶段，只能感觉“慢”
- 同一播放项重复打开时，解析等待会重复发生
- 首次起播会被字幕、弹幕、详情扩展等非关键工作阻塞
- 失败后用户需要自己判断该重试、换线路还是换解析器

## 方案选型

### 方案 A：只做 UI 提示

只补加载状态、错误提示和重试入口，不改播放链路。

优点：

- 风险最低
- 改动集中

缺点：

- 只能解释“为什么慢”，不能明显降低等待时间

### 方案 B：轻量链路优化

在现有播放模型上增加：

- 启动阶段状态显示
- 解析短时缓存
- 渐进式起播
- 失败后的快捷恢复动作

优点：

- 体感收益最大
- 不需要推倒现有播放器架构
- 适合逐步上线与回归验证

缺点：

- 会引入新的协调层
- 需要梳理播放器窗口和启动逻辑的边界

### 方案 C：重构为完整播放状态机

把详情打开、解析、起播、回退统一重构成显式状态机。

优点：

- 长期结构最完整
- 便于后续扩展自动降级、健康度排序、预取

缺点：

- 改动面太大
- 当前阶段回归风险高
- 不适合作为首次优化的第一步

### 结论

采用 **方案 B：轻量链路优化**。

原因：

- 直接命中首次播放慢的核心问题
- 可以在现有代码结构上增量实现
- 便于后续继续扩展为下一集预取、自动备用线路、来源健康度排序

## 总体设计

在现有 `controller -> request -> player window` 主流程上增加一层轻量播放启动协调，不改变现有页面控制器与播放器主职责。

核心思路：

- 播放前增加“可提前完成”的轻量准备
- 播放时先拿到最小可播 URL，尽快出首帧
- 其余附加信息异步补齐
- 失败时把可恢复动作直接呈现给用户

## 组件设计

### 1. PlaybackStartupCoordinator

新增轻量启动协调层，负责把用户点击播放转成统一的启动阶段流转。

职责：

- 管理启动阶段状态
- 先查缓存再决定是否解析
- 区分直链与解析型内容
- 产出统一失败类型与恢复动作建议

建议状态：

- `idle`
- `preparing`
- `resolving`
- `connecting`
- `buffering`
- `playing`
- `failed`

该组件不负责直接绘制 UI，也不取代现有播放器窗口。

### 2. PlaybackResolveCache

新增解析结果短时缓存层。

职责：

- 管理缓存键
- 管理 TTL
- 存取解析结果
- 区分不同解析器和不同播放项

首版仅缓存：

- 内置解析器结果

首版不缓存：

- 所有外部来源统一解析结果
- 复杂健康度数据

缓存键建议至少包含：

- 来源标识
- `vod_id` 或原始 URL
- 播放项/集数信息
- 解析器键

### 3. PlaybackWarmupService

新增轻量预热服务，用于播放前可提前执行的准备动作。

职责：

- 在详情页进入后，预热默认播放项
- 只做轻量准备，不直接播放
- 返回候选解析结果、失败摘要或预热命中信息

首版预热范围：

- 默认线路
- 默认播放项

首版不做：

- 相邻剧集预热
- 激进预拉流

### 4. PlayerWindow 扩展

`PlayerWindow` 保持 UI 和用户交互职责，但需要补充：

- 启动阶段状态显示
- 失败后的快捷动作按钮
- 渐进式补齐非关键内容
- 与解析器切换、线路切换的联动

`PlayerWindow` 不应承担缓存策略和回退决策本身。

## 数据流

### 正常首次起播

1. 用户在详情页点击播放项。
2. 请求进入 `PlaybackStartupCoordinator`。
3. 协调器进入 `preparing`。
4. 协调器查询 `PlaybackResolveCache`。
5. 若命中缓存：
   - 直接进入 `connecting`
6. 若未命中且当前项需要解析：
   - 进入 `resolving`
   - 完成后写入缓存
   - 再进入 `connecting`
7. 若是直链：
   - 直接进入 `connecting`
8. 建立连接后进入 `buffering`
9. 出首帧后进入 `playing`
10. 字幕、弹幕、详情扩展、插件动作异步补齐

### 失败路径

1. 任一阶段失败后进入 `failed`
2. 协调器生成失败类型
3. UI 根据当前上下文展示可恢复动作：
   - `重试`
   - `换线路`
   - `换解析器`

## UI 设计

### 启动状态显示

在播放器中增加轻量启动状态区，显示当前阶段与简短文案。

状态文案示例：

- `准备中`
- `解析中`
- `连接中`
- `缓冲中`
- `播放中`
- `播放失败`

说明文案示例：

- `正在准备播放项`
- `正在解析播放地址`
- `正在连接视频源`
- `正在等待首帧`
- `当前线路响应超时`
- `解析器未返回可播放地址`

设计要求：

- 不能遮挡核心播放控制
- 失败后状态区可复用为恢复动作入口
- 状态变化应与播放日志一致

### 渐进式起播

渐进式起播原则：

- 只要拿到最小可播 URL，就优先启动视频
- 不等待弹幕、字幕、扩展详情和插件动作全部完成
- 这些附加内容在视频起播后异步刷新

这样可以避免非关键内容阻塞首帧。

### 失败后的快捷动作

失败时直接展示最短恢复路径：

- `重试`
- `换线路`
- `换解析器`

展示规则：

- `重试`：始终可用
- `换线路`：仅多线路内容显示
- `换解析器`：仅解析型内容显示

第一版不做无提示的多次自动跳转，避免播放器行为不可控。

## 回退策略

首版采用三层恢复策略，但以“用户确认式恢复”为主。

### 第一层：同路径重试

适用场景：

- 网络短时抖动
- 代理连接失败
- 首包超时

行为：

- 保持当前线路和当前解析器
- 快速重试一次

### 第二层：切换备用线路

适用场景：

- 当前内容有多线路
- 当前线路失效或超时

行为：

- 提示用户切换下一条可用线路
- 第一版不做静默自动切换

### 第三层：切换备用解析器

适用场景：

- 当前内容属于解析型来源
- 当前解析器失败

行为：

- 提示用户切换到下一推荐解析器
- 第一版不做连续自动尝试多个解析器

## 第一阶段范围

第一阶段只实现最小闭环，优先拿到可见收益。

包含：

1. 启动状态可视化
2. 内置解析器短时缓存
3. 失败后的快捷动作

不包含：

1. 详情页预热
2. 自动备用线路
3. 自动备用解析器
4. 下一集预取
5. 来源健康度排序
6. 多来源统一缓存

职责边界：

  - 负责播放前轻量预热
- `playback_parsers.py`
  - 保持解析器能力提供者角色

## 验证策略

### 1. 状态流验证

验证不同来源下状态是否正确流转：

- `preparing -> resolving -> connecting -> buffering -> playing`
- `preparing -> connecting -> buffering -> playing`
- 任意阶段失败后进入 `failed`

### 2. 缓存验证

验证点：

- 同一播放项二次打开是否减少重复解析
- 缓存过期后是否正常重新解析
- 切换解析器后是否不会误命中旧缓存

### 3. 失败恢复验证

验证点：

- `重试` 是否可正常触发
- 多线路内容是否能正确显示 `换线路`
- 解析型内容是否能正确显示 `换解析器`

### 4. 渐进式起播验证

验证点：

- 视频能否先起播
- 弹幕、字幕、详情扩展是否可后补
- 失败状态和日志是否一致

重点覆盖的内容类型：

- 直链内容
- 解析型内容
- 多线路内容
- 恢复播放内容
- 带弹幕/字幕内容

## 迭代路线

### P0：首次播放加速

- 启动阶段可视化
- 解析短时缓存
- 详情页预热
- 渐进式起播

### P1：失败恢复与兜底

- 自动备用线路
- 自动备用解析器
- 错误分类提示
- 快捷重试面板增强

### P2：连续观看优化

- 下一集预取
- 换线路保进度
- 换清晰度保进度
- 系列级播放偏好

## 风险与约束

主要风险：

- `player_window.py` 已较大，继续堆逻辑会降低可维护性
- 状态显示和真实播放日志可能出现不一致
- 缓存键设计不当会导致误命中
- 渐进式起播若边界不清，可能引入字幕/弹幕延迟错觉

控制策略：

- 新能力尽量放入新增小模块
- 状态流转集中在协调层
- 缓存首版只覆盖内置解析器
- 每一阶段独立验证，不做大批量合并上线

## 结论

当前最合理的演进方式不是增加更多内容入口，而是优先重构“点击播放到首帧出现”这条链路的体感。

本设计选择以增量方式引入：

- 启动协调层
- 解析短时缓存
- 预热能力
- 渐进式起播
- 失败快捷恢复

并以“状态可见、等待可减、失败可恢复”为第一阶段标准，逐步向自动回退、预取和播放偏好记忆扩展。
