# Plugin Tab Overflow Navigation Design

## Summary

当插件 tab 数量过多时，主窗口顶部导航会出现标题截断、可发现性下降和切换成本升高的问题。这个设计保留现有顶部 tab 导航作为主路径，但只显示当前窗口宽度下能容纳的那部分插件 tab；放不下的插件收纳进一个“更多”入口，点击后从右侧滑出抽屉，并在抽屉中支持搜索和切换隐藏插件。

核心目标是优化导航体验，而不是重做插件信息架构。固定 tab 保持现状，插件页实例也保持常驻，布局变化只影响“哪些 tab 可见”和“哪些插件进入抽屉”，不触发插件页重建或重新加载。

## Goals

- 保留顶部 tab 导航作为插件主入口。
- 在宽度不足时，把溢出的插件 tab 收纳到“更多”入口后面。
- 用右侧抽屉展示隐藏插件，并支持快速搜索。
- 保持插件当前顺序，不因为导航优化而改变排序语义。
- 切换隐藏插件时不重建插件页实例、不刷新插件内容。
- 在窗口 resize 或插件集合变化后，正确重算可见/隐藏插件集合。

## Non-Goals

- 修改固定 tab 的顺序、命名或交互模型。
- 改造“插件管理”对话框的职责。
- 添加拖拽排序、收藏、置顶或分组管理。
- 把所有插件都移出顶部导航。
- 为溢出导航引入新的数据持久化配置。

## Scope

主要改动：

- `src/atv_player/ui/main_window.py`
- 可能新增一个主窗口内使用的轻量抽屉/面板控件文件，例如：
  - `src/atv_player/ui/plugin_tab_drawer.py`

主要验证：

- `tests/test_main_window_ui.py`

## Design

### Navigation Model

固定 tab 继续直接显示在顶部，行为不变。插件 tab 不再保证全部直接出现在顶部，而是按当前插件顺序依次尝试放入剩余宽度内。能放下的插件 tab 直接显示；放不下的插件进入隐藏集合，由一个统一的“更多”入口承载。

“更多”入口位于插件 tab 区域尾部，文案建议为：

- `更多`
- 当存在隐藏插件时显示数量，例如 `更多(12)`

没有隐藏插件时，不显示这个入口。

### Overflow Allocation

可见/隐藏插件划分规则：

1. 固定 tab 始终优先显示。
2. 计算固定 tab 占用宽度后，得到插件区剩余宽度。
3. 如果没有任何插件溢出，则插件 tab 按顺序全部显示，不显示“更多”入口。
4. 一旦有任意插件溢出，分配时必须预留“更多”入口本身的宽度。
5. 插件按当前顺序依次放入；第一个放不下的开始以及其后的插件全部进入隐藏集合。

这保证了：

- 可见插件仍然是“排在前面的常用插件优先可见”
- 隐藏插件集合稳定、可预测
- 不会因为宽度边界导致顶部最后一个插件和“更多”入口互相挤压

### More Drawer

点击“更多”入口后，从右侧滑出一个插件抽屉。抽屉不是新的顶层窗口，而是附着于主窗口的导航辅助面板。

抽屉行为：

- 宽度固定，避免遮满主内容区
- 只显示当前隐藏插件，不重复显示顶部已经可见的插件
- 列表顺序与插件当前顺序一致
- 当前激活但位于隐藏区的插件，在列表中高亮
- 点击列表项后：
  - 关闭抽屉
  - 切换主内容区到对应插件页
  - 不刷新页面数据
- `Esc` 可关闭抽屉
- 再次点击“更多”入口也可关闭抽屉

### Search Behavior

抽屉顶部有搜索框，打开抽屉后默认聚焦。

搜索规则：

- 基于插件标题做实时过滤
- 不改变原始排序
- 搜索只作用于隐藏插件集合，不包含顶部已经可见的插件
- 无匹配结果时显示空状态，如 `没有匹配的插件`

这样用户既能通过顶部快速访问常用插件，也能在插件很多时通过抽屉搜索快速定位长尾插件。

### Active Plugin Behavior

如果当前激活的插件属于隐藏集合：

- 顶部 tab 保持原样，不强行把它挪进可见区
- 抽屉列表中高亮当前项
- 主内容区继续正常显示该插件页

如果窗口变宽后，该插件重新进入顶部可见区：

- 抽屉中不再显示它
- 顶部 tab 直接承担当前激活状态

这能避免因窗口 resize 导致用户当前页面被打断或跳动。

### Recalculation Triggers

可见/隐藏插件集合只在以下时机重算：

- 主窗口首次构建插件 tab
- 插件集合发生结构性变化后重建插件 tab
- 主窗口 resize

普通切换 tab、打开播放器、执行页面加载，不应触发溢出导航重算。

### Page Lifecycle Constraint

这是本次设计的关键约束：布局变化只改变可见性分配，不重建插件页实例。

也就是说：

- `PosterGridPage` 仍然在插件 tab 初始化时创建
- resize 导致插件从顶部移入抽屉，或从抽屉回到顶部，不应重新创建页面对象
- 通过抽屉切换插件，也不应调用额外的插件 reload 流程

这可以避免把导航优化变成数据层或页面层的性能回退。

### Compatibility

现有行为应保持：

- 固定 tab 顺序和交互不变
- 插件 tab 顺序仍由现有插件定义顺序决定
- “插件管理”仍然是插件增删、启用、排序的唯一管理入口
- 当前 tab 持久化、分类记忆、插件页 `ensure_loaded()` 逻辑继续沿用

## Testing

需要覆盖：

- 窗口足够宽时，不显示“更多”入口
- 宽度不足时，只把溢出的插件放入抽屉
- resize 后可见/隐藏插件集合正确更新
- 点击抽屉插件项能切到对应插件页
- 抽屉搜索能过滤隐藏插件列表
- 当前激活的隐藏插件在抽屉中高亮
- 布局变化不会触发插件页重建或重新加载

## Open Questions

无。当前范围内的关键决策已固定：

- 顶部继续保留可见插件 tab
- 溢出插件进入“更多”右侧抽屉
- 抽屉支持搜索
- 只显示放不下的插件，不收纳全部插件
- 布局变化不重建插件页实例
