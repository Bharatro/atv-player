# Help Doc Release User Design

## Summary

重构 `docs/help.md` 的结构和叙述顺序，让它以“下载打包后的单体包并一键运行”的发行版用户为默认读者，同时补齐支持范围、功能依赖、技术特性、缓存迁移、术语约定和设置推荐值等参考信息。

本次只修改帮助文档内容与章节结构，不改 `README.md`、UI 文案、程序行为、截图资源或架构图。

## Goals

- 让终端用户在打开帮助文档前几分钟内就能完成“启动、登录、搜索、播放”的首次成功路径。
- 让文档默认匹配 GitHub Release 发行包的使用方式，而不是默认要求用户从源码运行。
- 明确 `yt-dlp` 是单独安装的可选外部依赖，只在播放相关页面链接时需要。
- 用集中章节说明支持的内容来源、协议范围和 `atv-player` 与 `alist-tvbox` 的职责边界。
- 补齐本地数据、缓存、重置、迁移和常见推荐值，减少后续重复提问。
- 统一“来源 / 来源分组 / 线路 / 剧集 / 解析器 / 播放项”等术语，降低阅读歧义。

## Non-Goals

- 不新增截图、界面标注图或架构图。
- 不同步修改 `README.md` 或发布页文案。
- 不改帮助对话框、快捷键、菜单入口或任何程序逻辑。
- 不承诺所有第三方网站、协议或插件在任何环境下都可用。
- 不把帮助文档写成开发者实现说明或 API 文档。

## Scope

主要改动：

- `docs/help.md`

本次不涉及测试代码或 UI 自动化变更，验证重点是文档结构、术语、可执行性和叙述准确性。

## Current Problem

当前 `docs/help.md` 已经具备较强的完整手册风格，但它仍然保留了明显的源码项目入口特征：

- 前部准备步骤默认从 `Python 3.12+`、`uv`、`libmpv` 开始，容易让发行版用户误以为这些也是一键运行包的前置条件。
- `yt-dlp` 的定位还不够明确，用户容易误解为它已经随应用打包，或者误以为所有播放都依赖它。
- 支持的网站、媒体协议、网盘和本地能力分散在各章节里，缺少集中索引。
- `atv-player` 本地能力、`alist-tvbox` 后端能力和外部工具依赖之间的边界没有专章说明。
- 数据目录已经写出，但“缓存为什么存在”“怎么重置”“怎么迁移到新电脑”还没有形成完整说明。
- “来源”“线路”“播放项”“解析”等概念在不同段落里混用，阅读时需要用户自己推断含义。

结果是：

- 发行版用户第一次阅读时门槛偏高。
- 用户更难快速判断某个问题属于播放器本体、后端能力、插件、`yt-dlp` 还是网络环境。
- 文档虽然完整，但还可以更接近“正式产品说明书”的阅读路径。

## Constraints

- 发布形态以 GitHub Release 中的打包后单体包为主，帮助文档必须优先服务这一使用方式。
- `yt-dlp` 不随应用一起打包，用户如需播放 `YouTube` 等页面链接，需要单独安装它。
- 当前默认播放缓存大小为 `512 MB`，普通流预读时长默认值为 `20` 秒；文档中的建议值不能与默认值冲突。
- 这轮改动只打磨可直接提交的文档结构与内容，不扩展到插图、README 或额外文档拆分。

## Approach Options

### Option A: Minimal insert-only update

只在现有章节中插入“快速开始”“支持范围”“依赖关系”等补充段落，尽量不重排章节。

优点：

- 改动小，风险低。

缺点：

- 文档整体阅读节奏仍然偏源码项目。
- 新内容更像后补条目，不会形成清晰的“上手层 + 参考层”分层。

### Option B: Re-layer the document for release users and reference readers

保留现有主线章节，但将前半部分改成“发行版用户首次成功路径”，把支持范围、依赖关系、技术特性、缓存迁移、术语约定集中到后半部分，形成一个清晰的参考信息层。

优点：

- 兼顾首次使用路径和成熟产品手册的参考价值。
- 可以复用大量现有内容，只重排章节并补关键缺口。
- 最符合“可直接提交的文档打磨”范围。

缺点：

- 需要调整章节编号和部分段落位置。

### Option C: Split into two documents

保留 `docs/help.md` 为操作手册，另起一份能力总览文档承接支持范围、技术特性和依赖关系。

优点：

- 信息分工更清楚。

缺点：

- 超出本次范围。
- 需要维护多份文档，并且会牵涉 README 或发布页链接调整。

## Decision

采用 **Option B**。

原因：

- 它能在不扩展文件数量的前提下，显著提升帮助文档对发行版用户的可读性。
- 它允许保留当前已经成熟的操作章节，只把“入口层”和“参考层”重新组织。
- 它最贴合这次目标：让 `docs/help.md` 更像可直接发布的产品说明书，而不是源码项目附带手册。

## Design

### 1. Reframe the reader and opening flow

文档开头默认面向发行版用户，而不是源码运行者。

开篇说明应保留“安装启动 -> 主窗口 -> 播放器 -> 直播源 -> 插件 -> 排障”的主线，但增加一句明确定位：

- 如果你使用的是 GitHub Release 提供的打包版本，可直接按“快速开始”完成首次使用。
- 只有从源码运行或自行打包时，才需要关注 Python、`uv` 和开发环境依赖。

### 2. Add a real quick start for packaged releases

新增“快速开始（3 分钟）”章节，放在“应用定位”之后。

这部分只写发行版用户最短路径：

1. 下载与你平台对应的发行包。
2. 解压。
3. 直接运行 `atv-player`。
4. 启动 `alist-tvbox` 后端。
5. 输入后端地址、用户名和密码登录。
6. 在顶部搜索框输入影片名。
7. 双击结果进入播放。

这里需要加两条边界说明：

- 如果要直接打开 `YouTube` 等页面链接，需要单独安装 `yt-dlp`。
- 如果你遇到安装、登录或播放问题，再继续阅读后文对应章节。

### 3. Split prerequisites into packaged users vs source users

现有“运行前准备”改为两段：

#### 发布版用户

只保留实际前置条件：

- 有一个可访问的 `alist-tvbox` 后端
- 系统网络环境可访问你要使用的内容来源
- 如需直接播放 `YouTube` 等页面链接，系统里需要单独安装可用的 `yt-dlp`

这部分不再要求发行版用户提前安装 Python、`uv` 或手动执行启动脚本。

#### 从源码运行

保留现有开发路径说明：

- Python `3.12+`
- `uv`
- 可用的 `libmpv`
- `uv sync --group dev`
- `./start.sh`
- `uv run src/atv_player/main.py`

并明确这是开发者、自构建用户或调试场景使用的路径。

### 4. Keep `yt-dlp` visible but bounded

`yt-dlp` 相关内容继续保留专题小节，但语气要更清晰：

- 它是外部工具，不随发行包一起提供。
- 它只在播放 `YouTube`、`X/Twitter`、`Instagram`、`TikTok` 等页面链接时参与工作。
- 安装了 `yt-dlp` 也不代表所有页面都一定能播放，结果仍受站点限制、网络和 Cookie 条件影响。

“快速开始”“运行前准备”“全局搜索直接打开”“`yt-dlp` 与 `YouTube` 播放”之间应形成一致表述，不重复堆砌细节。

### 5. Preserve the main usage path

以下主体章节保持现有主线和成熟内容，只做小幅润色、交叉引用和术语统一：

- 登录与启动
- 主窗口总览
- 各页面如何使用
- 播放器窗口
- 网络直播
- 插件管理
- 高级设置
- 快捷键总表
- 常见问题与排障

对这些章节的编辑原则：

- 不重写已成熟的功能说明。
- 只修正与发行版定位冲突的叙述。
- 只在必要处替换易混词汇。

### 6. Add a dedicated support matrix chapter

新增“支持的内容来源与协议”章节，放在“高级设置”之后、参考信息层之前部。

这一章使用分组总表，不深入讲操作：

- 视频网站：`YouTube`、B站、腾讯、爱奇艺、优酷、芒果等
- 媒体服务器：`Emby`、`Jellyfin`
- 网盘 / 分享链接：百度、阿里、夸克、`PikPak` 等
- 本地与远程媒体：普通 `HTTP/HTTPS` 媒体、`M3U8`、远程蓝光 `ISO`
- 协议与格式：`HLS`、`DASH`、磁力、`ed2k`、外挂字幕、`QRC/KRC/YRC`

必须补上一句范围声明：

- 实际可用性取决于当前后端能力、插件、网络环境和第三方站点状态。

### 7. Add a capability dependency chapter

新增“功能依赖关系”章节，直接回答“哪些能力属于播放器本地，哪些依赖后端或外部工具”。

建议分三类：

#### 仅本地播放器能力

- `mpv` 播放
- 主字幕 / 次字幕
- 弹幕搜索、切换、渲染和缓存
- 元数据刮削与展示
- 本地 HLS 代理
- 远程蓝光 `ISO` 流式读取
- 播放设置、日志、海报与本地缓存管理

#### 依赖 `alist-tvbox` 后端

- 登录
- 文件浏览
- 播放记录
- 网盘解析
- 离线下载
- 由后端能力驱动的标签页和媒体浏览

#### 依赖外部工具或外部环境

- `yt-dlp`
- 代理网络
- 浏览器 Cookie
- TMDB / Bangumi / 豆瓣等第三方服务配置
- 远程插件及其自身可用性

这一章要尽量写得像“问题归因指南”，而不是架构说明。

### 8. Add a player technical features chapter

新增“播放器技术特性”章节，定位是高级能力总览，而不是开发者内部设计。

建议覆盖：

- `mpv` 硬件解码
- `DASH` 多清晰度切换
- 多线路和自动切线
- 本地 HLS 代理与分片缓存
- 远程蓝光 `ISO` 流式读取
- `ASS` 弹幕渲染
- 主字幕 / 次字幕双轨
- `yt-dlp` 页面提取
- 卡拉 OK 歌词解析与渲染
- 播放恢复

每项用一句用户可理解描述，不引入内部模块名。

### 9. Expand local data into data, cache, reset, and migration guidance

现有“本地保存的数据”重构为“本地数据、缓存与迁移”。

建议包含四段：

#### 数据目录

保留现有标准路径说明：

- `~/.local/share/atv-player`
- `~/.cache/atv-player`

并说明不同平台目录位置可能不同，但应用使用的是 Qt 标准目录。

#### 会保存什么

保留并整理现有状态说明：

- 登录状态
- 插件配置与缓存
- 直播源和 EPG
- 播放历史与恢复信息
- 元数据绑定
- 弹幕偏好
- 主题、代理和播放设置

#### 缓存机制

集中说明会缓存：

- 海报
- 插件缓存
- 弹幕搜索结果
- 弹幕 XML / ASS
- 元数据结果
- HLS / TS 分片

并明确目的：

- 减少重复请求
- 提升加载速度
- 避免每次重新搜索、刮削或拉流

#### 重置与迁移

新增两个实际可执行的小节：

- 重置应用：删除数据目录和缓存目录即可恢复初始状态
- 迁移到新电脑：复制这两个目录即可尽量保留登录状态、插件、直播源、播放历史、元数据绑定和偏好设置

### 10. Add terminology conventions

新增“术语约定”章节，用短表统一常用词：

- `来源`
- `来源分组`
- `线路`
- `剧集`
- `解析器`
- `播放项`

该章节的职责是统一全文口径，而不是引入新概念。

### 11. Add user-facing recommendations under advanced settings

在“高级设置 -> 播放设置”下新增“普通用户建议”一小段，内容必须贴近当前默认值：

- 播放缓存大小默认 `512 MB`，普通用户通常保持默认即可
- 解码模式优先使用硬件解码
- 网络超时一般保持默认或按网络情况小幅调整
- 普通流预读时长默认 `20` 秒，普通用户通常保持默认即可
- 只有在卡顿、超时或兼容性异常时，再逐项调整

这里不要把建议写成硬性推荐，更不要写出与当前默认值冲突的数字。

### 12. Proposed chapter order

新的章节顺序应为：

1. 应用定位
2. 快速开始（3 分钟）
3. 运行前准备
4. 登录与启动
5. 主窗口总览
6. 各页面如何使用
7. 播放器窗口
8. 网络直播
9. 插件管理
10. 高级设置
11. 支持的内容来源与协议
12. 功能依赖关系
13. 播放器技术特性
14. 本地数据、缓存与迁移
15. 术语约定
16. 快捷键总表
17. 常见问题与排障

这个顺序的目标是：

- 先完成首次使用
- 再进入具体操作
- 最后再读能力索引、边界说明和参考信息

## Editing Plan

实施时对 `docs/help.md` 的修改方式应是：

1. 在开头新增“快速开始（3 分钟）”。
2. 改写“运行前准备”为“发布版用户 + 从源码运行”双路径。
3. 保留并微调现有 `yt-dlp` 说明，使其与发行版定位一致。
4. 在“高级设置”中补入贴近默认值的用户建议。
5. 在后半部分新增“支持的内容来源与协议”“功能依赖关系”“播放器技术特性”“术语约定”。
6. 将“本地保存的数据”重写为“本地数据、缓存与迁移”。
7. 对全文进行术语统一和交叉引用补强。

## Risks and Mitigations

### Risk: Overstating support scope

如果支持矩阵写得过满，用户会把它理解成绝对承诺。

Mitigation:

- 明确写出“取决于后端、插件、网络和第三方状态”。

### Risk: Mixing packaged and source workflows again

如果快速开始仍然掺入 `uv` 或 Python 依赖，发行版用户还是会被误导。

Mitigation:

- 快速开始只写下载、解压、运行、登录、搜索、播放。
- 源码路径单独放到“从源码运行”。

### Risk: Recommendations drifting from actual defaults

如果写错默认缓存或预读值，会削弱文档可信度。

Mitigation:

- 所有推荐值必须与当前应用默认值一致，当前已确认是 `512 MB` 和 `20` 秒。

## Result

完成后，`docs/help.md` 会更清晰地传达这些信息：

- 发行版用户如何在几分钟内完成首次使用
- `yt-dlp` 是单独安装的外部工具，而不是应用自带组件
- 应用支持哪些来源、协议和高级播放能力
- 哪些功能属于本地播放器，哪些依赖后端或外部环境
- 本地数据和缓存保存在哪里，为什么会占磁盘，如何重置和迁移
- 文档中的核心术语各自表示什么

整体效果应更接近正式发布版用户手册，而不是附带给源码仓库阅读的说明文档。
