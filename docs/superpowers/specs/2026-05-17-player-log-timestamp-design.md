# Player Log Timestamp Design

## Summary

为播放器右侧 `播放日志` 面板新增写入时的本地时间戳，格式固定为 `[HH:MM:SS.mmm] message`，用于提升播放排障时对解析、代理、弹幕、字幕和 `mpv` 事件的时间轴可读性。

本次改动只覆盖播放器窗口内部的 `播放日志` 文本输出，不调整全局 `logging` 配置，不引入分类、颜色或过滤能力。

## Goals

- 让播放器内每一条新日志都带毫秒级时间戳。
- 保持现有日志来源不变，包括普通 UI 路径、异步任务回调和弹幕日志 signal。
- 保持现有去重语义，避免同一条启动态日志因为时间戳不同而重复刷屏。
- 把改动限制在播放器日志写入层，尽量不扩散到大量调用点。

## Non-Goals

- 修改控制台或全局 `logging` 的格式。
- 在这一轮引入日志分类、颜色、高亮或过滤控件。
- 重写现有日志消息文案。
- 回填历史日志时间戳或迁移旧日志内容。

## Scope

主要改动：

- `src/atv_player/ui/player_window.py`

主要验证：

- `tests/test_player_window_ui.py`

## Design

### 1. Timestamp Injection Point

时间戳应在 `PlayerWindow._append_log(...)` 内统一注入，而不是分散到各个调用点。

原因：

- 现有播放器日志几乎都通过该入口写入。
- `self._danmaku_playback_log_signals.log` 也已经直接连接到该入口。
- 这样可以用最小改动覆盖解析、弹幕、字幕、代理、播放失败等现有日志来源。

统一入口会把传入的原始消息格式化为：

- `[22:07:13.421] 正在解析播放地址`
- `[22:07:14.002] 弹幕加载失败: ...`

时间戳使用写入当下的本地时间，精度到毫秒。

### 2. Dedupe Behavior

现有 `dedupe=True` 语义不能因为新增时间戳而失效。

如果仍然直接比较最终展示文本，那么同一条消息在两次写入时会因为毫秒不同而被视为不同内容，导致重复日志重新出现。为避免这个问题，去重应基于“原始消息内容”判断，而不是基于带时间戳后的最终字符串判断。

实现上应在 `PlayerWindow` 内保留一个最近一次写入的原始消息状态，仅用于连续去重判断：

- 当 `dedupe=False` 时，始终追加新行
- 当 `dedupe=True` 且最新一条原始消息与本次原始消息一致时，跳过写入
- 当消息不同，即使前缀时间戳可能相近，也正常追加

这样可以保持现有 resolving 启动态日志的抑制效果。

### 3. Compatibility Boundary

本次只改变播放器日志面板中的显示文本，不改变：

- `append_status_log(...)` 的调用方式
- 现有业务代码传给 `_append_log(...)` 的 message 内容
- 全局 `logging.basicConfig(...)` 的控制台输出格式

这意味着播放器 UI 日志和控制台日志会继续保持两条独立链路，只是在播放器面板中补齐时间戳。

### 4. Testing

补充 UI/单元混合层面的 focused 测试：

- `_append_log("播放失败: boom")` 后，`log_view` 文本符合 `^\[\d{2}:\d{2}:\d{2}\.\d{3}\] 播放失败: boom$`
- 连续两次对同一原始消息调用 `_append_log(..., dedupe=True)` 时，日志只保留一行
- 连续两次对不同消息调用 `_append_log(..., dedupe=True)` 时，日志保留两行，确保去重只抑制完全相同的原始消息

测试应继续使用现有 `PlayerWindow` 测试模式，不引入新的日志基础设施。

## Risks

- 如果去重仍然依赖最终展示行比较，新增时间戳后会破坏现有重复抑制。
- 如果时间戳生成散落到调用点，后续补分类时会更难统一演进。
- 如果测试只断言“包含消息文本”而不校验前缀格式，时间戳回归时不容易被发现。
